## 서비스 목록으로 운영체제의 표면 보기

2장은 운영체제 내부 알고리즘으로 바로 들어가기 전에, 운영체제가 바깥으로 무엇을 제공하는지 정리한다. 프로그램 실행, I/O, 파일 조작, 통신, 오류 감지 같은 기능은 사용자 편의를 위한 서비스다. 반면 자원 할당, 회계, 보호와 보안은 여러 사용자가 함께 쓰는 시스템을 안정적으로 운영하기 위한 서비스다.

이 구분이 중요한 이유는 운영체제가 단순히 편리한 라이브러리가 아니기 때문이다. 운영체제는 사용자 프로그램이 직접 해결하기 어려운 공유 자원 문제를 중앙에서 조정한다.

## API와 시스템 호출의 차이

프로그래머는 보통 `read()`, `write()`, `CreateProcess()` 같은 API를 호출한다. 하지만 API가 곧 시스템 호출은 아니다. 라이브러리는 인자를 준비하고, 필요한 경우 시스템 호출 번호와 함께 trap을 발생시켜 커널로 들어간다. 커널은 권한과 인자를 확인하고 실제 파일, 장치, 프로세스 작업을 수행한다.

따라서 시스템 호출은 사용자 모드와 커널 모드의 경계다. 이 경계를 기준으로 오류 처리 방식, 보안 검사, 성능 비용이 달라진다.

## 커널 구조의 선택

모놀리식 커널은 많은 기능을 커널 안에 두므로 호출 경로가 짧고 성능을 내기 쉽다. 대신 커널 내부 버그의 영향 범위가 넓다. 마이크로커널은 파일 시스템, 드라이버 같은 기능을 사용자 공간 서버로 옮겨 격리와 확장성을 노린다. 하지만 메시지 전달 비용과 구조 복잡성이 생긴다.

현대 운영체제는 한 가지 이름으로만 설명하기 어렵다. Linux는 단일 커널이지만 모듈을 동적으로 붙일 수 있고, Windows도 여러 계층과 서브시스템을 함께 사용한다. 여기서 볼 대목은 구조 이름을 외우는 것이 아니라, 성능·격리·유지보수 사이의 절충을 보는 것이다.

## 부팅과 디버깅까지 포함한 구현 관점

운영체제는 소스 코드만 있다고 실행되지 않는다. 펌웨어가 하드웨어를 초기화하고, 부트로더가 커널 이미지를 메모리에 올리며, 커널은 장치와 메모리, 첫 사용자 공간 프로세스를 준비한다. 여기까지가 되어야 사용자가 보는 로그인 화면이나 쉘이 나타난다.

또한 운영체제는 디버깅이 어렵다. 커널 오류는 시스템 전체 중단으로 이어질 수 있고, 동시성 버그는 재현이 까다롭다. 그래서 로그, 커널 디버거, crash dump, tracing 같은 도구가 운영체제 개발의 중요한 일부가 된다.

## 용어 정리

### 시스템 호출

- 의미: 응용 프로그램이 커널 기능을 요청하는 공식 인터페이스.
- 기능: 파일, 프로세스, 장치, 통신, 보호 기능을 커널에 맡긴다.
- 주의: API는 개발자가 보는 함수이고, 시스템 호출은 커널 경계로 들어가는 실제 요청이다.
- 키워드: `system call`, `API`

### 운영체제 서비스

- 의미: 프로그램 실행, I/O, 파일, 통신, 오류 감지 등 OS가 제공하는 기능.
- 기능: 응용 프로그램이 하드웨어 세부 사항 없이 실행되게 한다.
- 주의: 서비스는 사용자 편의 기능과 시스템 효율 기능으로 나눠 볼 수 있다.
- 키워드: `service`, `I/O`

### 커널 구조

- 의미: 커널 내부 기능을 배치하는 방식.
- 기능: 모놀리식, 계층형, 마이크로커널, 모듈형 구조로 성능과 격리를 조절한다.
- 주의: 마이크로커널은 작게 나누는 구조이지 항상 빠른 구조는 아니다.
- 키워드: `monolithic`, `microkernel`, `module`

### 부팅

- 의미: 펌웨어와 부트로더를 거쳐 커널을 메모리에 올리는 과정.
- 기능: 하드웨어 초기화 후 운영체제 실행 환경을 만든다.
- 주의: 부트로더는 커널이 아니라 커널을 적재하는 프로그램이다.
- 키워드: `boot loader`, `kernel image`

## 조심할 부분

- API와 시스템 호출을 같은 층으로 보면 사용자 라이브러리의 역할을 놓친다.
- 마이크로커널은 작게 나누는 구조이지 항상 더 빠른 구조라는 뜻은 아니다.
- 부트로더는 커널 자체가 아니라 커널을 메모리에 적재하는 프로그램이다.
