공룡책 10판 기준 PDF p.455 근처에서 시작한다.

## ==프로세스==마다 자기 ==메모리==가 있는 것처럼 보이게 하기

==프로세스==가 안전하게 실행되려면 CPU만 나눠 쓰는 것으로는 부족하다. 각 ==프로세스==가 자기 ==메모리==를 독립적으로 가진 것처럼 보여야 하고, 다른 ==프로세스==의 공간을 함부로 침범하지 못해야 한다. 그래서 ==논리 주소==와 ==물리 주소==를 분리한다.

처음에는 ==연속 할당==으로 설명할 수 있지만, 시간이 지나면 빈 공간이 잘게 흩어진다. ==페이징==은 이 문제를 고정 크기 조각으로 바꿔 푼다. 대신 페이지 테이블과 ==TLB==라는 주소 변환 비용이 생기므로, ==메모리== 관리는 배치 문제이면서 성능 문제다.

## ==논리 주소==에서 ==TLB==까지 한 번에 보기

### 주소 공간은 ==운영체제==가 만들어 주는 착시다

프로그램은 자기 ==메모리==가 0번지부터 이어진다고 생각한다. 실제 RAM에서는 여러 ==프로세스==의 페이지가 뒤섞여 있을 수 있다. ==MMU==와 페이지 테이블이 이 차이를 가려 주기 때문에 ==프로세스==마다 독립된 주소 공간을 가진 것처럼 실행된다.

### ==연속 할당==은 단순하지만 오래 버티기 어렵다

==프로세스==를 물리 ==메모리==에 통째로 연속 배치하면 base와 limit만으로 ==보호==를 설명할 수 있어 직관적이다. 문제는 ==프로세스==가 들어오고 나간 뒤 빈 공간이 잘게 흩어진다는 점이다. compaction은 가능하지만 비용이 크고, 실행 중인 ==메모리==를 옮기는 부담이 만만치 않다.

### ==페이징==은 단편화와 변환 비용의 거래다

==페이징==은 외부 단편화를 줄이고 비연속 배치를 가능하게 한다. 대신 페이지 테이블을 저장해야 하고, 주소 변환을 매번 처리해야 한다. ==TLB==가 빠른 이유는 최근 변환 결과를 기억해 이 비용을 줄이기 때문이다.

### 페이지 테이블 항목에는 주소만 있지 않다

페이지 테이블은 ==page== number를 ==frame== number로 바꾸는 표지만, 실제 항목에는 valid bit, protection bit, dirty bit, reference bit 같은 정보도 들어간다. 주소 변환, ==보호==, 교체 정책이 이 한 항목에 같이 걸려 있다. 그래서 ==메모리== 관리는 단순 배치 문제가 아니라 ==보호==와 성능 정책의 교차점이다.

### ==TLB==는 빠르지만 문맥 교환과 만난다

==TLB==에는 최근 주소 변환 결과가 들어 있다. 그런데 문맥 교환이 일어나면 같은 가상 주소라도 다른 프로세스에서는 다른 ==물리 주소==를 뜻할 수 있다. 그래서 ==TLB==를 비우거나, ASID처럼 프로세스 식별 정보를 붙여 구분한다. 주소 변환 캐시도 ==스케줄링== 비용과 연결된다.

## ==연속 할당==에서 ==페이징==까지 따라가기

### 주소 공간을 분리하는 이유

프로그램은 자신이 0번지부터 시작하는 연속된 ==메모리==를 쓰는 것처럼 생각한다. 이것이 ==논리 주소== 공간이다. 실제 RAM의 위치는 ==물리 주소==이고, 둘 사이의 변환은 ==MMU==와 ==운영체제==가 함께 처리한다. 이 분리 덕분에 프로세스마다 독립된 주소 공간을 제공할 수 있다.

주소 변환은 ==보호==에도 필요하다. 어떤 프로세스가 자기 범위를 벗어난 주소에 접근하면 하드웨어와 ==커널==이 이를 막아야 한다. 메모리 관리는 성능 기능이면서 동시에 ==보호== 기능이다.

![메인 메모리 - 주소 변환 위치](/os-concepts/diagrams/os-09-main-memory-detail.png)

> 그림 읽기: CPU가 만든 주소가 곧 RAM 주소는 아니다. ==MMU==와 ==TLB==, 페이지 테이블을 거쳐야 물리 프레임을 찾을 수 있다.

### ==연속 할당==과 단편화

초기의 단순한 방식은 프로세스 하나를 물리 메모리의 연속된 공간에 넣는 것이다. base와 limit 레지스터만 있어도 주소 변환과 ==보호==를 구현할 수 있다. 하지만 프로세스가 들어오고 나가면 빈 공간이 여기저기 나뉘어 외부 단편화가 생긴다.

compaction으로 빈 공간을 모을 수 있지만, 실행 중인 프로세스의 메모리를 옮기려면 비용이 크고 주소 재배치가 필요하다. 그래서 현대 시스템은 고정 크기 조각으로 나누는 ==페이징==을 기본으로 사용한다.

### ==페이징==을 보는 기준

==페이징==은 논리 메모리를 ==page==로, 물리 메모리를 ==frame==으로 나눈다. ==논리 주소==는 ==page== number와 offset으로 나뉘고, 페이지 테이블은 ==page== number를 ==frame== number로 바꾼다. offset은 그대로 유지된다. 이렇게 하면 프로세스의 페이지들이 물리 메모리 곳곳에 흩어져 있어도 연속된 주소 공간처럼 보인다.

외부 단편화는 줄지만 페이지 테이블 크기와 내부 단편화가 생긴다. 페이지 크기가 커지면 테이블은 작아지지만 내부 단편화가 커지고, 페이지 크기가 작아지면 테이블과 변환 비용이 커질 수 있다.

![메인 메모리 - 페이징 주소 변환](/os-concepts/diagrams/os-09-main-memory.png)

> 그림 읽기: ==논리 주소==의 ==page== number는 페이지 테이블을 통해 ==frame== number로 바뀌고 offset은 그대로 따라간다. 이 구조 덕분에 논리적으로는 연속, 물리적으로는 비연속 배치가 가능하다.

### ==TLB==와 다단계 페이지 테이블

메모리를 접근할 때마다 페이지 테이블을 메모리에서 읽으면 너무 느리다. TLB는 최근 주소 변환 결과를 저장하는 작은 고속 캐시다. TLB hit이면 바로 ==물리 주소==를 만들 수 있고, miss이면 페이지 테이블을 따라가야 한다.

주소 공간이 커질수록 페이지 테이블 전체를 단순 배열로 두기 어렵다. 다단계 페이지 테이블, inverted ==page== table 같은 구조는 메모리 사용량을 줄이기 위한 절충이다. 주소 변환은 단순 공식이 아니라 캐시와 자료구조의 조합이다.

## 가상 주소 하나가 RAM 주소가 되기까지

1. CPU가 만든 주소는 먼저 페이지 번호와 오프셋으로 나뉜다.
2. TLB에 변환 결과가 있으면 바로 프레임 번호를 얻고, 없으면 페이지 테이블을 따라간다.
3. 페이지 테이블 항목이 유효하면 프레임 번호와 오프셋을 합쳐 ==물리 주소==가 된다. 유효하지 않다면 10장의 ==page== fault로 이어진다.

## 메모리 장에서 놓치면 안 되는 말들

- **==논리 주소==**: 프로그램이 생성하는 주소. 프로세스마다 독립된 주소 공간을 갖게 한다. 다만 ==논리 주소==는 실제 RAM 위치가 아니다. 주요 키워드: `logical address`.
- **==물리 주소==**: 메모리 장치가 실제로 접근하는 주소. ==MMU==가 ==논리 주소==를 변환해 만든다. 다만 ==운영체제==가 ==보호==와 배치를 관리한다. 주요 키워드: `physical address`, `MMU`.
- **==연속 할당==**: 프로세스를 물리 메모리의 연속된 공간에 배치하는 방식. 단순하지만 외부 단편화가 생긴다. 다만 compaction은 비용이 크다. 주요 키워드: `contiguous allocation`.
- **==페이징==**: 논리 메모리와 물리 메모리를 고정 크기 조각으로 나누는 방식. 외부 단편화를 줄이고 비연속 배치를 가능하게 한다. 다만 내부 단편화와 페이지 테이블 비용은 남는다. 주요 키워드: `paging`, `page`, `frame`.
- **TLB**: 페이지 번호와 프레임 번호 매핑을 저장하는 빠른 캐시. 주소 변환 비용을 줄인다. 다만 데이터 캐시가 아니라 주소 변환 캐시다. 주요 키워드: `TLB`, `hit ratio`.

9장이 현재 메모리에 올라온 페이지를 다뤘다면, 10장은 필요한 페이지만 올리는 ==가상 메모리==로 확장한다.
