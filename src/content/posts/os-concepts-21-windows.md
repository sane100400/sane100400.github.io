## ==Windows==를 GUI가 아니라 NT 구조로 읽기

==Windows== 장은 ==Linux==와 다른 이름과 구조로 같은 ==운영체제== 문제를 푸는 사례다. GUI가 아니라 NT 계열 구조를 보면 사용자 모드와 ==커널== 모드, ==Executive==, Kernel, HAL, 드라이버가 어떻게 역할을 나누는지 보인다.

==Windows==의 중요한 특징은 객체와 핸들 모델이다. 파일, ==프로세스==, 이벤트 같은 ==커널== 자원을 객체로 보고, 사용자 프로그램은 포인터가 아니라 핸들로 접근한다. ==보안 참조 모니터==는 접근 토큰과 ==보안== 설명자를 비교해 이 접근이 허용되는지 판단한다.

## ==Executive==, Kernel, HAL을 역할로 나눠 보기

### ==Windows==는 GUI보다 NT 구조로 봐야 한다

사용자가 보는 화면은 ==Windows==의 일부일 뿐이다. NT 계열 ==Windows==는 사용자 모드와 ==커널== 모드를 나누고, ==커널== 모드 안에 ==Executive==, Kernel, HAL, 드라이버를 둔다. 이 계층을 봐야 ==프로세스==, ==메모리==, I/O, ==보안==이 어디에서 처리되는지 보인다.

### ==Executive==와 Kernel은 같은 말이 아니다

==Executive==는 객체, ==프로세스==, ==메모리==, I/O, ==보안== 같은 고수준 서비스를 제공한다. Kernel은 ==스레드== ==스케줄링==, ==인터럽트==, 예외, ==동기화== primitive처럼 더 낮은 수준의 일을 맡는다. HAL은 하드웨어 차이를 감춰 이식성을 돕는다.

### 핸들 모델은 ==보호==와 자원 회수를 쉽게 만든다

사용자 프로그램은 ==커널== 객체 포인터를 직접 받지 않고 핸들을 받는다. ==커널==은 핸들 테이블과 ==보안== 설명자를 보고 접근을 허용할지 판단한다. 이 방식은 권한 검사, 참조 카운트, 객체 회수를 한곳에서 관리하기 좋다.

### 객체 모델은 ==Windows==의 공통 언어다

Windows는 파일, ==프로세스==, 이벤트, 세마포어를 모두 객체로 다룬다. 객체마다 이름, 타입, 참조 카운트, ==보안== 설명자가 붙고, ==프로세스==는 핸들로 접근한다. 이 공통 모델 덕분에 서로 다른 자원도 비슷한 방식으로 권한 검사와 수명 관리를 받을 수 있다.

### 토큰과 ==보안== 설명자를 같이 봐야 한다

접근 토큰에는 사용자의 SID와 그룹, 권한 정보가 들어 있고, 객체의 ==보안== 설명자에는 누가 어떤 접근을 할 수 있는지가 적혀 있다. ==보안 참조 모니터==는 요청된 접근 권한을 이 둘과 비교한다. Windows ==보호== 모델의 중심 흐름은 이 검사 과정이다.

## ==커널== 구조에서 ==보안 참조 모니터==까지 따라가기

### Windows를 구조로 보기

Windows는 GUI만으로 이해하기 어렵다. NT 계열 Windows는 사용자 모드와 ==커널== 모드를 나누고, 커널 모드 안에 ==Executive==, Kernel, device driver, HAL 같은 구성 요소를 둔다. 사용자 모드에는 Win32 API와 서비스, 환경 서브시스템이 있다.

이 구조의 목표는 호환성, ==보안==, 이식성, 성능을 함께 달성하는 것이다. 앞 장의 ==Linux==와 비교하면 같은 ==운영체제== 문제를 다른 용어와 구조로 해결한다는 점이 보인다.

![Windows 시스템 - Windows 커널 모드 구조](/os-concepts/diagrams/os-21-windows.png)

> 그림 읽기: Windows 커널 모드에는 ==Executive==, Kernel, HAL, 여러 관리자가 계층적으로 놓인다. 객체, 보안, I/O, ==메모리== 관리가 이 구조 안에서 연결된다.

### ==Executive==와 Kernel

Executive는 ==프로세스==, ==스레드==, ==메모리==, I/O, 객체, 보안 같은 고수준 ==운영체제== 서비스를 제공한다. Kernel은 더 낮은 수준에서 ==스레드== ==스케줄링==, ==인터럽트==와 예외 처리, ==동기화== primitive 같은 중심 기능을 담당한다.

HAL은 하드웨어 차이를 감추는 계층이다. 커널과 드라이버가 특정 보드나 장치 세부 사항에 덜 묶이도록 돕는다. 이 구분이 있어 이식성과 유지보수를 챙기기 쉽다.

### 객체와 핸들

Windows는 파일, ==프로세스==, ==스레드==, 이벤트, 세마포어 같은 커널 자원을 객체로 다룬다. ==객체 관리자==는 객체 이름, 참조 카운트, 핸들 테이블을 관리한다. 사용자 프로그램은 커널 객체를 직접 가리키는 포인터가 아니라 핸들을 통해 접근한다.

핸들 기반 모델은 ==보호==와 자원 회수에 유리하다. ==프로세스==마다 핸들 테이블이 있고, 커널은 핸들로 접근 권한을 확인한다.

![Windows 시스템 - Windows 객체 접근 검사](/os-concepts/diagrams/os-21-windows-detail.png)

> 그림 읽기: 프로세스가 객체를 열면 ==객체 관리자==가 ==보안 참조 모니터==에 검사를 맡기고, 허용된 경우 핸들을 돌려준다. 사용자 코드는 커널 포인터가 아니라 핸들로 객체를 다룬다.

### 보안과 ==파일 시스템==

==보안 참조 모니터==는 접근 토큰과 보안 설명자를 이용해 객체 접근을 검사한다. 사용자가 로그인하면 토큰에 SID와 권한 정보가 담기고, 객체에는 누가 무엇을 할 수 있는지 나타내는 보안 설명자가 붙는다. 접근 요청은 이 정보를 비교해 허용되거나 거부된다.

==NTFS==는 Windows의 대표 ==파일 시스템==으로 ==MFT==, 권한, 저널링, 압축, 링크 등을 제공한다. Windows 사례를 보면 ==보호==, ==파일 시스템==, 객체 모델이 서로 분리된 주제가 아니라 하나의 실행 환경 안에서 연결되어 있음을 알 수 있다.

## Windows에서 핸들로 객체에 접근할 때

1. 프로세스가 파일이나 이벤트 객체를 열면 ==객체 관리자==가 커널 객체를 찾는다.
2. ==보안 참조 모니터==는 접근 토큰과 보안 설명자를 비교해 허용 여부를 판단한다.
3. 허용되면 프로세스는 핸들을 받고, 이후에는 그 핸들을 통해 객체를 사용한다.

## Windows 내부 구조를 읽는 말들

- **==Windows 구조==**: 사용자 모드와 커널 모드, Executive, Kernel, HAL로 나뉜 구조. 호환성, 보안, 이식성, 성능을 계층적으로 처리한다. 다만 Windows를 GUI만으로 이해하면 핵심을 놓친다. 주요 키워드: `Windows`, `Executive`.
- **Executive**: 커널 모드의 고수준 관리자 집합. 프로세스, ==메모리==, I/O, 객체, 보안 관리를 담당한다. 다만 Kernel보다 더 높은 수준의 OS 서비스를 제공한다. 주요 키워드: `Executive`.
- **==객체 관리자==**: 커널 객체의 이름, 핸들, 수명을 관리하는 구성 요소. 파일, 프로세스, 이벤트 같은 자원을 일관된 방식으로 다룬다. 다만 핸들은 사용자 공간 포인터가 아니라 커널 객체 참조다. 주요 키워드: `object manager`, `handle`.
- **==보안 참조 모니터==**: 접근 토큰과 보안 설명자를 바탕으로 접근을 검사하는 구성 요소. Windows ==보호== 모델의 중심 경로다. 다만 권한 검사는 객체 접근 시점에 반복적으로 일어난다. 주요 키워드: `access token`, `security descriptor`.
- **==NTFS==**: Windows의 대표 ==파일 시스템==. ==MFT==, 권한, 저널링, 압축, 링크, change journal을 제공한다. 다만 저널링은 모든 사용자 데이터를 완전히 보존한다는 뜻이 아니다. 주요 키워드: `NTFS`, `MFT`.

21장까지 보면 공룡책에서 다루는 큰 범위는 한 번 훑은 셈이다. 이후에는 각 장의 연습 문제나 실제 ==Linux==/Windows 관찰로 이해를 굳히는 편이 좋다.
