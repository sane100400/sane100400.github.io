## 프로그램이 프로세스가 되는 순간

디스크에 저장된 실행 파일은 프로그램이고, 실행을 시작해 주소 공간과 CPU 상태를 얻으면 프로세스가 된다. 프로세스에는 코드뿐 아니라 현재 실행 위치, 레지스터 값, 스택, 힙, 열린 파일, 권한 정보가 포함된다. 운영체제는 이 묶음을 하나의 작업 단위로 관리한다.

그래서 같은 프로그램을 두 번 실행하면 보통 서로 다른 프로세스 두 개가 생긴다. 코드 파일은 같아도 주소 공간, 스택, 파일 디스크립터, 실행 위치가 다르기 때문이다.

## 상태 전이와 PCB

프로세스는 생성된 뒤 바로 CPU를 쓰는 것이 아니라 ready queue에서 기다린다. 스케줄러가 선택하면 running 상태가 되고, I/O를 요청하면 waiting 상태로 내려간다. I/O가 끝나면 다시 ready 상태가 된다. 이 상태 전이는 운영체제가 CPU를 여러 프로세스 사이에 나눠 주는 기본 그림이다.

문맥 교환이 일어나면 커널은 현재 프로세스의 레지스터, 프로그램 카운터, 상태를 PCB에 저장하고 다음 프로세스의 PCB를 읽어 복원한다. 문맥 교환 자체는 유용한 일을 계산하지 않으므로 비용이지만, 멀티태스킹을 가능하게 하는 필수 비용이다.

## 프로세스 생성과 실행 이미지 교체

UNIX 계열에서는 `fork()`가 현재 프로세스를 복제하고, `exec()`가 복제된 프로세스의 실행 이미지를 새 프로그램으로 바꾼다. 이 모델 덕분에 쉘은 명령을 입력받아 자식 프로세스를 만들고, 그 자식이 실제 프로그램으로 바뀌어 실행된다.

부모는 자식의 종료를 `wait()`로 회수해야 한다. 회수되지 않은 종료 프로세스는 zombie가 될 수 있고, 부모가 먼저 사라진 자식은 init 또는 systemd 같은 상위 프로세스가 맡는다.

## IPC는 격리와 협력의 균형

프로세스는 기본적으로 서로의 주소 공간을 침범하지 못한다. 이 격리는 안정성에는 좋지만 협력에는 통신 수단이 필요하다. 공유 메모리는 같은 물리 메모리를 여러 프로세스가 매핑해 빠르게 데이터를 주고받는 방식이다. 대신 동시에 접근할 때 동기화 문제가 생긴다.

메시지 전달은 커널이 메시지를 중개하므로 구조가 명확하고 보호가 쉽다. 하지만 복사와 시스템 호출 비용이 들 수 있다. 책은 이 두 방식의 차이를 보여 주면서 이후 동기화 장으로 자연스럽게 연결한다.

## 용어 정리

### 프로세스

- 의미: 실행 중인 프로그램.
- 기능: CPU 스케줄링, 메모리 보호, 파일 소유의 기본 단위가 된다.
- 주의: 프로그램은 파일이고 프로세스는 실행 상태까지 포함한다.
- 키워드: `process`, `address space`

### PCB

- 의미: 프로세스 상태를 저장하는 커널 자료구조.
- 기능: 문맥 교환 때 레지스터, PC, 상태, 열린 파일 정보를 저장하고 복원한다.
- 주의: PCB는 사용자 프로그램이 직접 건드릴 수 없다.
- 키워드: `PCB`, `context switch`

### 프로세스 상태

- 의미: new, ready, running, waiting, terminated 같은 실행 단계.
- 기능: 스케줄러가 어떤 프로세스를 실행할 수 있는지 판단하게 한다.
- 주의: waiting은 CPU가 없어서가 아니라 I/O나 이벤트를 기다리는 상태다.
- 키워드: `ready`, `waiting`

### IPC

- 의미: 프로세스 사이의 통신 방법.
- 기능: 공유 메모리나 메시지 전달로 독립된 프로세스가 협력하게 한다.
- 주의: 공유 메모리는 빠르지만 동기화가 필요하다.
- 키워드: `IPC`, `shared memory`, `message passing`

## 조심할 부분

- 프로그램은 수동적인 파일이고, 프로세스는 실행 상태와 자원 소유 정보를 가진 동적인 객체다.
- waiting은 CPU 차례를 기다리는 상태가 아니라 I/O나 이벤트 완료를 기다리는 상태다.
- 공유 메모리는 빠르지만, 빠르다는 이유만으로 동기화가 사라지는 것은 아니다.
