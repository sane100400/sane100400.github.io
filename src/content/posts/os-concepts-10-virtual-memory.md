## 프로그램 전체를 꼭 ==메모리==에 올려야 할까

==메모리==를 배웠다면 다음 질문은 프로그램 전체가 꼭 ==메모리==에 있어야 하느냐다. 실제 실행은 특정 순간에 필요한 일부 페이지만 집중적으로 쓰는 경우가 많다. ==가상 메모리==는 이 성질을 이용해 필요한 페이지를 늦게 가져온다.

하지만 늦게 가져오는 전략은 ==page fault==라는 비용을 만든다. 페이지가 없으면 ==커널==이 디스크를 다녀오고, 빈 프레임이 없으면 누군가를 내보내야 한다. ==페이지 교체==와 ==working set==은 결국 ==메모리==가 부족할 때 무엇을 남기고 무엇을 버릴지 정하는 이야기다.

## ==요구 페이징==과 교체 정책을 연결하기

### ==가상 메모리==는 게으른 적재 전략이다

프로그램 전체를 처음부터 ==메모리==에 올리지 않고, 실제로 접근한 페이지를 그때그때 가져온다. 이 덕분에 큰 프로그램도 빨리 시작할 수 있고 여러 ==프로세스==를 함께 실행하기 쉽다. 다만 필요한 순간 디스크를 다녀와야 하므로 ==page fault==가 많아지면 장점이 바로 사라진다.

### ==page fault==는 예외 처리와 I/O가 한꺼번에 온다

==page fault==가 나면 ==커널==은 단순히 비트를 하나 바꾸고 끝내지 않는다. 접근이 합법인지 보고, 빈 프레임을 찾거나 희생 페이지를 고르고, 필요한 경우 dirty page를 디스크에 쓰고, 새 페이지를 읽어 온다. 마지막에는 중단된 명령을 다시 실행해야 한다.

### 교체 정책은 지역성에 기대고 있다

==LRU== 계열 정책은 최근에 쓴 페이지를 곧 다시 쓸 가능성이 높다는 지역성에 기대고 있다. 정확한 ==LRU==는 비싸서 clock 같은 근사 기법이 등장한다. ==working set==은 ==프로세스==가 한동안 실제로 만진 페이지를 보고 필요한 프레임 규모를 가늠하는 방식이다.

### dirty bit가 교체 비용을 가른다

희생 페이지를 고를 때 그 페이지가 수정되지 않았다면 그냥 버려도 된다. 하지만 dirty bit가 켜져 있으면 디스크에 다시 써야 한다. 같은 page replacement라도 읽기만 한 페이지와 수정된 페이지의 비용은 다르다. 교체 정책은 fault 수뿐 아니라 쓰기 비용도 함께 고려해야 한다.

### ==가상 메모리==는 공유에도 쓰인다

==가상 메모리==는 큰 주소 공간을 보여 주는 장치일 뿐 아니라 공유 라이브러리, memory-mapped file, copy-on-write fork에도 쓰인다. 서로 다른 ==프로세스==의 가상 페이지가 같은 물리 프레임을 가리키게 만들 수 있기 때문이다. 주소 변환은 격리와 공유를 동시에 가능하게 한다.

## ==page fault==에서 ==thrashing==까지 따라가기

### ==가상 메모리==의 출발점

==가상 메모리==는 프로그램 전체가 물리 ==메모리==에 있어야 실행된다는 생각을 버린다. 실제로 실행에 필요한 부분은 특정 시점의 일부 페이지뿐이다. 나머지는 디스크에 있다가 접근될 때 ==메모리==로 올라올 수 있다. 이 방식은 큰 프로그램 실행, 여러 ==프로세스== 동시 실행, ==프로세스== 간 공유를 쉽게 만든다.

다만 ==가상 메모리==는 무한 ==메모리==가 아니다. 물리 메모리와 저장장치 사이를 오가며 착시를 만드는 구조다. ==page fault==가 잦아지면 성능은 급격히 떨어진다.

### ==요구 페이징==의 흐름

페이지 테이블에는 해당 페이지가 메모리에 있는지 나타내는 valid-invalid bit가 있다. ==프로세스==가 invalid 페이지에 접근하면 ==page fault== trap이 발생한다. ==커널==은 접근이 합법적인지 확인하고, 빈 frame을 찾은 뒤, 디스크에서 페이지를 읽어 오고, 페이지 테이블을 갱신한 다음 중단된 명령을 다시 실행한다.

이 흐름은 I/O를 포함하므로 일반 메모리 접근보다 훨씬 비싸다. 그래서 ==page fault== rate를 낮추는 것이 ==가상 메모리== 성능을 크게 좌우한다.

![가상 메모리 - 요구 페이징 처리](/os-concepts/diagrams/os-10-virtual-memory.png)

> 그림 읽기: valid bit가 꺼진 페이지에 접근하면 page fault가 난다. ==커널==은 합법성 검사, 프레임 확보, 디스크 읽기, 페이지 테이블 갱신을 거쳐 중단된 명령을 다시 실행한다.

![가상 메모리 - Page fault 비용](/os-concepts/diagrams/os-10-virtual-memory-detail.png)

> 그림 읽기: page fault 비용은 trap만이 아니라 디스크 I/O를 포함한다. 그래서 ==페이지 교체== 정책과 ==working set== 관리가 성능에 직접 영향을 준다.

### ==페이지 교체== 정책

빈 frame이 없으면 어떤 페이지를 내보낼지 정해야 한다. ==FIFO==는 오래 들어와 있던 페이지를 내보내지만, 자주 쓰이는 페이지도 오래됐다는 이유로 쫓아낼 수 있다. Optimal은 앞으로 가장 늦게 사용될 페이지를 내보내므로 기준점으로는 좋지만 미래를 알아야 하므로 실제 구현은 어렵다.

==LRU==는 최근에 사용되지 않은 페이지가 앞으로도 덜 쓰일 것이라는 지역성에 기대는 정책이다. 정확한 ==LRU== 구현은 비용이 크기 때문에 reference bit, clock algorithm 같은 근사 기법이 자주 쓰인다.

### ==Thrashing==과 ==working set==

==프로세스==에 할당된 frame이 작업 집합보다 적으면 필요한 페이지를 계속 쫓아내고 다시 가져오게 된다. 이때 CPU는 계산보다 ==페이지 교체==와 디스크 I/O에 시간을 쓰며, 시스템 전체 처리량이 떨어진다. 이것이 ==thrashing==이다.

==working set== model은 일정 시간 창 안에서 실제로 사용한 페이지 집합을 추적해 ==프로세스==가 필요한 메모리 규모를 추정한다. 프로세스 수를 무작정 늘리는 것이 항상 좋은 일은 아니라는 점도 봐야 한다.

## 처음 접근한 페이지가 메모리에 없을 때

1. CPU가 invalid 페이지를 참조하면 하드웨어가 trap을 걸고 ==커널==로 들어간다.
2. ==커널==은 접근이 합법인지 확인한 뒤 빈 프레임을 찾거나 희생 페이지를 고른다.
3. 디스크에서 페이지를 읽고 페이지 테이블을 고친 다음, 중단됐던 명령을 다시 실행한다. 그래서 page fault는 일반 메모리 접근보다 훨씬 비싸다.

## 가상 메모리를 설명하는 핵심 말들

- **가상 메모리**: 물리 메모리보다 큰 주소 공간을 쓰는 것처럼 보이게 하는 기법. 프로그램 일부만 메모리에 올려 메모리 사용 효율을 높인다. 다만 무한 메모리를 제공하는 기술은 아니다. 주요 키워드: `virtual memory`.
- **==요구 페이징==**: 접근한 페이지가 필요할 때만 메모리에 올리는 방식. 초기 적재 비용과 메모리 사용량을 줄인다. 다만 page fault가 자주 나면 성능이 크게 떨어진다. 주요 키워드: `demand paging`.
- **==페이지 폴트==**: 접근한 페이지가 메모리에 없을 때 발생하는 트랩. ==커널==이 페이지를 디스크에서 읽어 오도록 만든다. 다만 일반 캐시 미스보다 훨씬 비싸다. 주요 키워드: `page fault`.
- **==페이지 교체==**: 빈 프레임이 없을 때 내보낼 페이지를 고르는 작업. ==FIFO==, ==LRU==, Optimal 같은 정책으로 fault 수를 줄인다. 다만 Optimal은 실제 구현보다 비교 기준이다. 주요 키워드: `FIFO`, `LRU`.
- **==Thrashing==**: 계산보다 ==페이지 교체==에 시간을 더 쓰는 상태. ==working set==과 page-fault frequency로 감지한다. 다만 프로세스를 더 넣으면 오히려 악화될 수 있다. 주요 키워드: `thrashing`, `working set`.
