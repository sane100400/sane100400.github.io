## 스레드가 필요한 이유

프로세스는 주소 공간을 가진 큰 실행 단위다. 프로세스 두 개는 코드가 같더라도 서로 다른 주소 공간을 쓰고, 한쪽의 스택이나 힙을 다른 쪽이 바로 만질 수 없다. 그래서 협력하려면 파이프, 메시지 큐, 공유 메모리 같은 IPC가 필요하다.

스레드는 같은 프로세스 안에서 움직이는 더 작은 실행 흐름이다. 코드, 데이터, 힙, 열린 파일은 함께 쓰고, 레지스터와 스택은 스레드마다 따로 둔다. 덕분에 생성과 전환 비용이 줄고 같은 데이터에 접근하기 쉽지만, 공유가 쉬운 만큼 잘못된 접근도 쉬워진다. 이 지점에서 동기화 문제가 시작된다.

## 병행성과 병렬성을 구분하기

병행성은 여러 작업이 진행 중인 것처럼 구성되는 성질이다. 단일 코어에서도 운영체제가 실행 흐름을 빠르게 바꿔 가며 병행성을 만들 수 있다. 병렬성은 실제로 같은 순간 여러 코어에서 작업이 실행되는 성질이다.

멀티코어 환경에서는 병렬성을 얻을 기회가 생기지만 자동으로 성능이 좋아지지는 않는다. 작업을 어떻게 나눌지, 데이터 의존성을 어떻게 줄일지, 캐시와 동기화 비용을 어떻게 관리할지가 더 중요해진다.

## 사용자 스레드와 커널 스레드

사용자 스레드는 라이브러리가 관리하므로 생성과 전환이 빠를 수 있다. 하지만 커널이 그 존재를 모르면 하나의 사용자 스레드가 블로킹 시스템 호출을 했을 때 같은 프로세스의 다른 사용자 스레드까지 멈출 수 있다. 커널 스레드는 커널이 스케줄링하므로 멀티코어 병렬 실행과 블로킹 처리가 자연스럽다.

many-to-one, one-to-one, many-to-many 모델은 사용자 스레드와 커널 스레드를 어떻게 연결하느냐의 차이다. 현대 범용 시스템은 보통 one-to-one 모델을 많이 사용한다.

![스레드와 병행성 - 사용자 스레드와 커널 스레드](/os-concepts/diagrams/os-04-threads-concurrency.png)

> 그림 읽기: 사용자 스레드가 많아도 커널 스레드에 어떻게 연결되는지에 따라 블로킹과 병렬 실행이 달라진다. 결국 CPU에 올리는 단위는 커널이 스케줄링할 수 있는 실행 흐름이다.

## 스레드 프로그래밍의 현실적인 문제

스레드를 많이 만들면 요청마다 대응하기 쉽지만, 스레드 생성 비용과 메모리 사용량이 커진다. thread pool은 미리 만들어 둔 스레드가 작업 큐에서 일을 가져가게 해 이 비용을 줄인다. fork/join은 큰 작업을 작은 작업으로 나누고 결과를 합치는 병렬 프로그래밍 모델이다.

책은 스레드 취소, signal 처리, thread-local storage 같은 세부 문제도 다룬다. 공통점은 하나다. 실행 흐름이 여러 개가 되면 제어 흐름과 자원 정리가 생각보다 복잡해진다.

## 예시: 웹 서버가 여러 요청을 처리할 때

1. 요청마다 프로세스를 새로 만들면 주소 공간도 따로 생기므로 격리는 좋지만 비용이 크다.
2. 스레드를 쓰면 같은 캐시, 설정, 열린 소켓을 공유하면서 요청을 나누어 처리한다.
3. 대신 여러 스레드가 같은 요청 큐나 통계 카운터를 건드릴 수 있으므로 락과 조건 변수가 필요해진다.

## 용어 정리

### 스레드

- 의미: 프로세스 안의 실행 흐름.
- 기능: 같은 주소 공간을 공유하면서 여러 작업을 동시에 진행한다.
- 주의: 스택과 레지스터는 스레드마다 따로 있고, 힙과 파일은 보통 공유한다.
- 키워드: `thread`, `stack`

### 병행성

- 의미: 여러 작업이 겹쳐 진행되는 성질.
- 기능: 대기 시간이 있는 작업을 효율적으로 섞어 실행한다.
- 주의: 단일 코어에서도 병행성은 가능하다.
- 키워드: `concurrency`

### 병렬성

- 의미: 여러 작업이 같은 순간 실제로 동시에 실행되는 성질.
- 기능: 멀티코어에서 계산량을 나눠 처리한다.
- 주의: 스레드가 많다고 항상 병렬성이 늘지는 않는다.
- 키워드: `parallelism`, `multicore`

### 스레드 모델

- 의미: 사용자 스레드와 커널 스레드를 연결하는 방식.
- 기능: many-to-one, one-to-one, many-to-many로 블로킹과 병렬성 특성이 달라진다.
- 주의: 커널이 스케줄링하는 대상은 결국 커널 스레드다.
- 키워드: `user thread`, `kernel thread`

## 조심할 부분

- 스레드는 프로세스보다 작지만, 아무 비용이 없는 실행 단위는 아니다.
- 병행성은 구조의 문제이고 병렬성은 실제 동시 실행의 문제다.
- 주소 공간 공유는 편리함과 경쟁 조건을 동시에 만든다.

## 이어지는 내용

여러 프로세스와 스레드가 생기면 CPU를 누구에게 줄지가 문제가 된다. 5장은 스케줄링을 다룬다.
