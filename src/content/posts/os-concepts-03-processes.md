## 프로그램은 언제 ==프로세스==가 되는가

프로그램은 디스크에 놓인 파일이지만, ==프로세스==는 실행 중인 활동이다. 같은 프로그램을 두 번 실행하면 코드 파일은 같아도 실행 위치, 스택, 열린 파일, 권한 정보가 서로 다르다. ==운영체제==는 이 실행 중인 묶음을 하나의 관리 단위로 본다.

==프로세스==가 생기면 곧 CPU를 나눠 쓰는 문제가 생긴다. 실행 중이던 ==프로세스==를 멈추고 다른 ==프로세스==를 올리려면 현재 상태를 어딘가에 저장해야 하고, 그 역할을 ==PCB==가 맡는다. 독립된 주소 공간 덕분에 안전해진 대신, ==프로세스==끼리 협력하려면 ==IPC==라는 별도의 통로가 필요해진다.

## 상태, ==PCB==, ==IPC==를 한 흐름으로 보기

### ==프로세스==는 실행 중인 흔적까지 포함한다

프로그램 파일만으로는 현재 어디까지 실행했는지 알 수 없다. ==프로세스==에는 프로그램 카운터, 레지스터 값, 스택, 힙, 열린 파일, 권한 정보가 붙는다. ==운영체제==는 이 묶음을 보고 중단했던 실행을 다시 이어 가고, 다른 프로세스와 주소 공간을 분리한다.

### ==PCB==는 ==커널==이 남기는 책갈피다

문맥 교환은 단순히 이름표만 바꾸는 일이 아니다. ==커널==은 실행 중이던 프로세스의 CPU 상태를 ==PCB==에 저장하고, 다음 프로세스의 상태를 꺼내 CPU에 올린다. 이 작업 자체는 사용자의 계산을 진행시키지 않지만, 여러 프로세스가 한 CPU를 나눠 쓰려면 피할 수 없는 비용이다.

### 격리된 프로세스가 협력하려면 통로가 필요하다

프로세스는 서로의 ==메모리==를 마음대로 읽지 못한다. 이 격리 덕분에 안전하지만, 함께 일하려면 ==IPC==가 필요하다. 공유 ==메모리==는 빠른 대신 ==동기화== 책임이 커지고, 메시지 전달은 구조가 분명한 대신 ==커널== 개입과 복사 비용이 생긴다.

### ==ready==와 ==waiting==을 정확히 나누기

==ready== 상태는 CPU만 받으면 바로 실행할 수 있는 상태다. ==waiting== 상태는 I/O 완료, 자식 종료, 메시지 도착 같은 사건이 먼저 필요하다. 이 둘을 섞으면 스케줄러가 무엇을 고를 수 있는지 헷갈린다. 스케줄러는 ==ready== queue에서만 고르고, ==waiting== queue의 프로세스는 사건이 끝나야 다시 ==ready==로 돌아온다.

### fork, exec, wait가 나눠진 이유

UNIX 계열에서 `fork()`는 실행 흐름을 복제하고, `exec()`는 프로그램 이미지를 갈아 끼우며, `wait()`는 부모가 자식의 종료 상태를 회수하게 한다. 셋을 분리하면 쉘이 리다이렉션이나 파이프를 설정한 뒤 자식만 다른 프로그램으로 바꾸는 식의 유연한 실행 구성이 가능해진다.

## 프로세스 생성부터 통신까지 따라가기

### 프로그램이 프로세스가 되는 순간

디스크에 저장된 실행 파일은 프로그램이고, 실행을 시작해 주소 공간과 CPU 상태를 얻으면 프로세스가 된다. 프로세스에는 코드뿐 아니라 현재 실행 위치, 레지스터 값, 스택, 힙, 열린 파일, 권한 정보가 포함된다. ==운영체제==는 이 묶음을 하나의 작업 단위로 관리한다.

그래서 같은 프로그램을 두 번 실행하면 보통 서로 다른 프로세스 두 개가 생긴다. 코드 파일은 같아도 주소 공간, 스택, 파일 디스크립터, 실행 위치가 다르기 때문이다.

### 상태 전이와 ==PCB==

프로세스는 생성된 뒤 바로 CPU를 쓰는 것이 아니라 ==ready== queue에서 기다린다. 스케줄러가 선택하면 running 상태가 되고, I/O를 요청하면 ==waiting== 상태로 내려간다. I/O가 끝나면 다시 ==ready== 상태가 된다. 이 상태 전이는 ==운영체제==가 CPU를 여러 프로세스 사이에 나눠 주는 기본 그림이다.

문맥 교환이 일어나면 ==커널==은 현재 프로세스의 레지스터, 프로그램 카운터, 상태를 ==PCB==에 저장하고 다음 프로세스의 ==PCB==를 읽어 복원한다. 문맥 교환 자체는 유용한 일을 계산하지 않으므로 비용이지만, 멀티태스킹을 가능하게 하는 필수 비용이다.

![프로세스 - 프로세스 상태 전이](/os-concepts/diagrams/os-03-processes.png)

> 그림 읽기: ==ready==와 ==waiting==을 구분해야 한다. ==ready==는 CPU만 기다리는 상태이고, ==waiting==은 I/O나 이벤트가 끝나야 다시 실행 후보가 되는 상태다.

### 프로세스 생성과 실행 이미지 교체

UNIX 계열에서는 `fork()`가 현재 프로세스를 복제하고, `exec()`가 복제된 프로세스의 실행 이미지를 새 프로그램으로 바꾼다. 이 모델 덕분에 쉘은 명령을 입력받아 자식 프로세스를 만들고, 그 자식이 실제 프로그램으로 바뀌어 실행된다.

부모는 자식의 종료를 `wait()`로 회수해야 한다. 회수되지 않은 종료 프로세스는 zombie가 될 수 있고, 부모가 먼저 사라진 자식은 init 또는 systemd 같은 상위 프로세스가 맡는다.

![프로세스 - fork와 exec](/os-concepts/diagrams/os-03-processes-detail.png)

> 그림 읽기: 쉘이 명령을 실행할 때 fork는 실행 흐름을 복제하고 exec는 자식의 프로그램 이미지를 바꾼다. 이 둘을 나눠 보는 것이 UNIX식 프로세스 생성을 이해하는 기준이 된다.

### ==IPC==는 격리와 협력의 균형

프로세스는 기본적으로 서로의 주소 공간을 침범하지 못한다. 이 격리는 안정성에는 좋지만 협력에는 통신 수단이 필요하다. 공유 ==메모리==는 같은 물리 ==메모리==를 여러 프로세스가 매핑해 빠르게 데이터를 주고받는 방식이다. 대신 동시에 접근할 때 ==동기화== 문제가 생긴다.

메시지 전달은 ==커널==이 메시지를 중개하므로 구조가 명확하고 ==보호==가 쉽다. 하지만 복사와 ==시스템 호출== 비용이 들 수 있다. 책은 이 두 방식의 차이를 보여 주면서 이후 ==동기화== 장으로 자연스럽게 연결한다.

## 쉘에서 명령 하나를 실행할 때

1. 사용자가 쉘에 `ls`를 입력하면 쉘은 먼저 자식 프로세스를 만든다.
2. 자식은 `exec()`로 자신의 실행 이미지를 `ls` 프로그램으로 바꾼다. 부모인 쉘은 `wait()`로 자식 종료를 기다린다.
3. 이 과정에서 ==PCB==, ready queue, 문맥 교환, 종료 상태 회수가 한 번씩 등장한다.

## 프로세스를 설명할 때 필요한 말들

- **프로세스**: 실행 중인 프로그램. CPU ==스케줄링==, ==메모리== ==보호==, 파일 소유의 기본 단위가 된다. 다만 프로그램은 파일이고 프로세스는 실행 상태까지 포함한다. 주요 키워드: `process`, `address space`.
- **PCB**: ==프로세스 상태==를 저장하는 ==커널== 자료구조. 문맥 교환 때 레지스터, PC, 상태, 열린 파일 정보를 저장하고 복원한다. 다만 PCB는 사용자 프로그램이 직접 건드릴 수 없다. 주요 키워드: `PCB`, `context switch`.
- **==프로세스 상태==**: new, ready, running, ==waiting==, terminated 같은 실행 단계. 스케줄러가 어떤 프로세스를 실행할 수 있는지 판단하게 한다. 다만 ==waiting==은 CPU가 없어서가 아니라 I/O나 이벤트를 기다리는 상태다. 주요 키워드: `ready`, `waiting`.
- **==IPC==**: 프로세스 사이의 통신 방법. 공유 ==메모리==나 메시지 전달로 독립된 프로세스가 협력하게 한다. 다만 공유 ==메모리==는 빠르지만 ==동기화==가 필요하다. 주요 키워드: `IPC`, `shared memory`, `message passing`.

프로세스 안에서 더 작은 실행 흐름을 여러 개 두면 ==스레드==가 된다. 4장은 이 차이를 다룬다.
