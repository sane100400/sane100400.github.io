## 앞에서 배운 개념이 ==Linux==에서 만나는 지점

==Linux== 장은 앞에서 배운 개념이 실제 ==커널==에서 어떻게 만나는지 확인하는 사례다. ==프로세스==와 ==스레드==는 task로 다뤄지고, ==메모리== 관리는 페이지와 page cache를 통해 파일 I/O와 연결된다. ==VFS==는 다양한 ==파일 시스템==을 같은 인터페이스로 묶는다.

==Linux==는 단일 ==커널== 구조지만 모듈과 공통 인터페이스로 확장성을 확보한다. ==CFS==, copy-on-write ==fork==, namespace, cgroup, capability 같은 기능을 보면 ==스케줄링==, ==가상 메모리==, ==보호==가 따로 떨어진 주제가 아니라는 점이 드러난다.

## ==Linux==를 단일 ==커널==과 모듈 구조로 읽기

### ==Linux==는 앞 장 개념이 한곳에 모인 사례다

==Linux==를 보면 ==프로세스==, ==스케줄링==, ==메모리==, ==파일 시스템==, 네트워크, ==보호==가 따로 놀지 않는다. ==시스템 호출==은 사용자 공간에서 ==커널== 공간으로 들어오는 입구이고, ==커널== 안에서 task, page, inode, socket 같은 객체가 서로 맞물린다.

### 단일 ==커널==이지만 확장 지점이 있다

==Linux==는 많은 기능이 ==커널== 공간에서 함께 도는 단일 ==커널== 구조다. 그렇다고 모든 기능을 항상 고정해 두는 것은 아니다. loadable ==kernel module==로 드라이버나 ==파일 시스템==을 붙일 수 있고, ==VFS==와 네트워크 스택처럼 공통 인터페이스도 갖고 있다.

### 컨테이너는 ==보호== 기능의 조합으로 이해하면 쉽다

namespace는 ==프로세스==가 보는 이름 공간을 나누고, cgroup은 CPU와 ==메모리== 같은 자원 사용량을 제한한다. capability와 LSM은 권한을 더 잘게 조절한다. 컨테이너는 별도 커널을 띄우는 VM이라기보다 이런 ==Linux== 기능을 묶어 격리된 실행 환경을 만드는 쪽에 가깝다.

### task_struct로 ==프로세스==와 ==스레드==를 함께 본다

Linux는 ==프로세스==와 ==스레드==를 모두 task 관점으로 다룬다. 주소 공간, 파일 테이블, signal 처리 같은 자원을 얼마나 공유하느냐에 따라 ==프로세스==처럼 보이기도 하고 ==스레드==처럼 보이기도 한다. 이 관점은 `fork()`, `clone()`, thread 구현을 함께 이해하게 해 준다.

### ==CFS==는 공평함을 시간으로 계산한다

Completely Fair Scheduler는 각 작업이 CPU를 얼마나 받았는지 가상 실행 시간으로 추적한다. 덜 실행된 작업이 더 먼저 선택될 가능성이 커진다. 단순히 우선순위 숫자만 보는 것이 아니라, 전체 CPU 시간을 어떻게 나눌지 계속 보정하는 방식이다.

## ==프로세스==, ==메모리==, ==VFS==, ==보안== 기능 따라가기

### Linux를 사례로 보는 이유

Linux는 UNIX 계열 아이디어를 바탕으로 한 오픈소스 ==운영체제==이고, 서버, 임베디드, 모바일, 클라우드에서 폭넓게 쓰인다. 책의 앞 장에서 배운 ==프로세스==, ==메모리==, ==파일 시스템==, 네트워크, ==보호== 개념을 실제 커널 구조로 연결해 보기 좋은 사례다.

Linux는 모듈을 지원하지만 마이크로커널은 아니다. 중심 서비스는 커널 공간에서 함께 동작하고, 필요한 기능을 loadable ==kernel module==로 확장할 수 있다.

![Linux 시스템 - Linux 시스템 구조](/os-concepts/diagrams/os-20-the-linux-system.png)

> 그림 읽기: Linux는 사용자 공간과 커널 공간을 나누고, 커널 안에서 스케줄러, ==메모리== 관리자, ==VFS==, 네트워크 스택, 드라이버가 함께 동작한다.

### 프로세스와 ==스케줄링==

Linux는 프로세스와 ==스레드==를 모두 task라는 관점으로 관리한다. `fork()`는 새 task를 만들고, `exec()`는 실행 이미지를 교체한다. copy-on-write는 ==fork== 직후 모든 ==메모리==를 즉시 복사하지 않고, 실제 쓰기가 발생할 때 복사해 비용을 줄인다.

==스케줄링==은 일반 작업과 실시간 작업을 구분한다. Completely Fair Scheduler는 일반 작업이 CPU를 공정하게 나눠 쓰도록 가상 실행 시간을 사용한다. 실시간 정책은 더 높은 우선순위 규칙을 가진다.

![Linux 시스템 - copy-on-write fork](/os-concepts/diagrams/os-20-the-linux-system-detail.png)

> 그림 읽기: copy-on-write ==fork==는 부모와 자식이 처음에는 같은 물리 페이지를 공유하다가 쓰기가 발생한 페이지만 복사한다. 그래서 ==fork== 직후 비용을 크게 줄일 수 있다.

### ==메모리==와 ==파일 시스템==

Linux 메모리 관리는 페이지 기반 ==가상 메모리==, demand paging, page cache, slab 계열 allocator를 사용한다. 파일을 읽을 때도 디스크 블록은 page cache를 거쳐 재사용될 수 있다. 그래서 파일 I/O와 메모리 관리는 서로 깊게 연결되어 있다.

==VFS==는 ext4, XFS, tmpfs, ==procfs==, NFS 같은 ==파일 시스템==을 공통 인터페이스로 묶는다. `/proc`은 실제 디스크 파일이라기보다 커널 상태를 파일처럼 보여 주는 가상 ==파일 시스템==이다.

### I/O, 네트워크, ==보안==

Linux의 장치 드라이버는 ==커널 모듈==로 제공되는 경우가 많고, 블록 장치와 문자 장치를 구분해 다룬다. 네트워크 스택은 소켓 인터페이스를 거쳐 사용자 프로그램과 연결되고, 패킷 처리 경로는 드라이버와 프로토콜 계층을 지난다.

==보안== 측면에서는 사용자와 그룹 권한, capability, namespace, cgroup, LSM 같은 기능이 조합된다. 컨테이너 기술은 이 중 namespace와 cgroup을 적극적으로 사용한다.

## Linux에서 ==fork==가 가벼워 보이는 이유

1. `fork()` 직후 부모와 자식은 같은 물리 페이지를 공유한다. 둘 중 하나가 쓰기 전까지 실제 복사는 미뤄진다.
2. 쓰기 발생 시 해당 페이지만 복사하므로 명령 실행 준비가 빠르다.
3. 이 copy-on-write 전략은 프로세스 생성, ==가상 메모리==, 페이지 ==보호==가 함께 맞물린 사례다.

## Linux 구조를 읽는 말들

- **==Linux 커널==**: UNIX 계열 아이디어를 바탕으로 한 모듈형 단일 커널. 프로세스, 메모리, 파일, 네트워크, 장치 관리를 수행한다. 다만 모듈을 쓴다고 마이크로커널은 아니다. 주요 키워드: `Linux`, `kernel`.
- **==커널 모듈==**: 실행 중인 커널에 동적으로 붙는 코드. 드라이버, ==파일 시스템==, 네트워크 기능을 확장한다. 다만 커널 권한으로 실행되므로 버그 영향이 크다. 주요 키워드: `kernel module`.
- **==fork/exec==**: UNIX식 프로세스 생성과 프로그램 교체 모델. 쉘, 파이프, 서버 프로세스 생성의 기본이 된다. 다만 ==fork== 후 메모리는 copy-on-write로 지연 복사될 수 있다. 주요 키워드: `fork`, `exec`.
- **==Linux 스케줄링==**: 일반 작업과 실시간 작업을 구분해 CPU를 배정하는 구조. 공정성과 반응성을 함께 고려한다. 다만 정책별 우선순위 의미가 다르다. 주요 키워드: `CFS`, `real-time`.
- **==Linux VFS==**: 여러 파일 시스템을 공통 인터페이스로 묶는 계층. ext4, ==procfs==, tmpfs, NFS를 같은 API로 다루게 한다. 다만 /proc은 디스크 파일이 아니라 커널 상태를 파일처럼 보여 준다. 주요 키워드: `VFS`, `procfs`.

Linux 사례를 봤다면, 21장에서는 ==Windows== NT 계열 구조가 같은 문제를 다른 방식으로 푸는 모습을 본다.
