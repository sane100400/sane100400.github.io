`Operating System Concepts` 10판 Chapter 12, **I/O Systems** 정리다.

공룡책 10판 기준 PDF p.617 근처에서 시작한다. 장치 컨트롤러, 포트, 버스, 드라이버의 역할부터 polling, interrupt, DMA의 차이까지 차례로 정리한다.

## 먼저 볼 부분

## I/O는 표준화된 인터페이스와 장치별 구현의 조합

응용 프로그램은 파일처럼 읽고 쓰는 단순한 인터페이스를 원한다. 하지만 실제 장치는 키보드, 디스크, 네트워크 카드처럼 동작 방식이 다르다. 커널 I/O 서브시스템은 공통 인터페이스를 제공하고, 장치 드라이버는 특정 장치의 명령과 상태 처리를 담당한다.

장치 컨트롤러는 하드웨어 쪽에서 장치를 제어하고, 드라이버는 커널 쪽에서 컨트롤러를 다룬다. 이 구분을 잡으면 I/O 경로가 명확해진다.

## polling, interrupt, DMA

polling은 CPU가 장치 상태를 반복해서 확인하는 방식이다. 구현은 단순하지만 장치가 느리면 CPU 시간이 낭비된다. interrupt는 장치가 완료나 오류를 CPU에게 알려 주는 방식이다. CPU는 다른 일을 하다가 필요한 순간에 커널 처리 루틴으로 이동한다.

DMA는 대량 데이터 전송에서 CPU가 바이트를 하나씩 복사하지 않게 한다. 장치 컨트롤러가 메모리와 직접 데이터를 주고받고, 완료되면 인터럽트로 알린다. 고성능 I/O에서 DMA는 핵심 장치다.

## 커널 I/O 서브시스템

커널은 I/O 요청을 단순 전달만 하지 않는다. 버퍼링으로 속도가 다른 장치 사이를 맞추고, 캐싱으로 반복 접근을 줄이며, 스풀링으로 독점 장치에 대한 요청을 큐에 쌓는다. 오류가 발생하면 재시도하거나 사용자에게 오류를 전달한다.

I/O 보호도 중요하다. 사용자 프로그램이 장치 레지스터를 직접 건드리면 시스템이 불안정해지므로, 대부분의 장치 접근은 시스템 호출과 드라이버가 제한한다.

## 성능은 경로 전체에서 결정된다

I/O 요청은 응용 프로그램, 시스템 호출, 커널 I/O 계층, 드라이버, 컨트롤러, 장치를 거친다. 어느 한 곳만 빠르다고 전체가 빠르지는 않다. 작은 요청이 너무 자주 발생하면 시스템 호출과 컨텍스트 전환 비용이 커지고, 큰 요청은 버퍼와 DMA 설정 비용을 고려해야 한다.

운영체제의 I/O 설계는 추상화와 성능 사이의 절충이다. 사용자는 단순한 파일 인터페이스를 원하지만, 커널은 장치 특성을 최대한 활용해야 한다.

## 용어 정리

### 장치 드라이버

- 의미: 커널과 장치 컨트롤러 사이의 장치별 코드.
- 기능: 장치 명령과 상태 처리를 운영체제 인터페이스에 맞춘다.
- 주의: 드라이버 버그는 커널 안정성에 직접 영향을 줄 수 있다.
- 키워드: `driver`

### 장치 컨트롤러

- 의미: 실제 장치를 제어하는 하드웨어 구성 요소.
- 기능: 레지스터와 버퍼를 통해 CPU나 DMA와 통신한다.
- 주의: 드라이버와 컨트롤러는 소프트웨어/하드웨어로 구분된다.
- 키워드: `controller`

### 폴링

- 의미: CPU가 장치 상태를 반복 확인하는 방식.
- 기능: 단순한 장치나 짧은 대기에서 쓸 수 있다.
- 주의: 대기 시간이 길면 CPU를 낭비한다.
- 키워드: `polling`

### 인터럽트

- 의미: 장치가 CPU에게 처리 필요를 알리는 방식.
- 기능: CPU가 I/O 완료를 기다리며 놀지 않게 한다.
- 주의: 너무 잦으면 interrupt overhead가 커진다.
- 키워드: `interrupt`

### DMA

- 의미: 장치와 메모리 사이의 데이터 전송을 CPU 대신 처리하는 방식.
- 기능: 대량 I/O에서 CPU 복사 부담을 줄인다.
- 주의: 메모리 보호와 캐시 일관성을 함께 봐야 한다.
- 키워드: `DMA`

## 조심할 부분

- 드라이버는 소프트웨어이고 컨트롤러는 장치를 제어하는 하드웨어다.
- 인터럽트가 항상 polling보다 좋은 것은 아니다. 매우 짧고 잦은 작업에서는 오버헤드가 커질 수 있다.
- DMA를 쓰면 CPU 복사는 줄지만 메모리 보호와 캐시 일관성 문제를 함께 봐야 한다.
