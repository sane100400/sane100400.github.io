## 파일 이름은 어떻게 실제 블록을 찾는가

파일 인터페이스가 사용자에게 보이는 얼굴이라면, ==파일 시스템 구현==은 그 뒤에서 이름과 메타데이터와 데이터 블록을 연결하는 일이다. 디렉터리 엔트리는 이름을 내부 식별자에 연결하고, ==inode==나 ==FCB==는 실제 데이터 블록 위치를 가리킨다.

파일이 커지고 지워지고 다시 만들어지면 공간 배치 문제가 생긴다. ==연속 할당==은 빠르지만 자라기 어렵고, ==연결 할당==은 확장하기 쉽지만 바로 찾아가기 어렵다. ==색인 할당==과 빈 공간 관리, 저널링은 ==파일 시스템==이 성능과 복구 가능성을 함께 다루는 방식이다.

## 할당 방식과 복구 문제를 함께 보기

### 파일 구현은 이름, 메타데이터, 데이터 블록을 잇는 일이다

디렉터리 엔트리는 파일 이름을 내부 식별자에 연결한다. ==inode==나 ==FCB==는 파일 크기, 권한, 시간 정보, 데이터 블록 위치를 담는다. 사용자가 파일 이름을 입력하면 ==운영체제==는 이 여러 단계를 따라 실제 블록까지 도착한다.

### 할당 방식은 성장과 임의 접근에서 갈린다

==연속 할당==은 빠르고 단순하지만 파일이 커질 때 주변 공간이 비어 있어야 한다. ==연결 할당==은 확장이 쉽지만 중간 위치로 바로 가기 어렵다. ==색인 할당==은 인덱스 블록을 두어 임의 접근을 쉽게 만들지만 인덱스 관리 비용이 든다.

### 빈 공간 관리와 복구도 ==파일 시스템==의 일부다

파일을 저장하는 일만큼 빈 블록을 추적하는 일도 필요하다. ==bitmap==이나 free list는 어떤 블록이 비어 있는지 기록한다. 시스템이 쓰기 도중 멈추면 메타데이터와 데이터가 서로 어긋날 수 있어 journaling이나 consistency check가 등장한다.

### ==inode== 포인터 구조를 파일 크기와 연결하기

UNIX 계열 ==inode==는 직접 포인터, 단일 간접 포인터, 이중 간접 포인터 같은 구조로 작은 파일과 큰 파일을 함께 처리한다. 작은 파일은 빠르게 직접 블록을 찾고, 큰 파일은 간접 블록을 거쳐 더 많은 데이터 블록을 가리킨다. 파일 크기가 커질수록 탐색 단계가 늘어나는 이유다.

### 저널링은 순서를 기록하는 기술이다

==파일 시스템==은 데이터와 메타데이터를 여러 위치에 쓴다. 중간에 전원이 꺼지면 일부만 반영되어 구조가 깨질 수 있다. 저널링은 변경 의도를 먼저 로그에 남기고 실제 구조를 갱신해, 재부팅 뒤 어느 지점까지 반영됐는지 판단하게 해 준다.

## ==inode==에서 journaling까지 따라가기

### ==파일 시스템==은 여러 층으로 구현된다

사용자는 파일 이름과 read/write만 보지만, 내부에는 논리 ==파일 시스템==, 파일 조직 모듈, 기본 ==파일 시스템==, I/O 제어 계층이 있다. 위쪽은 이름과 메타데이터를 다루고, 아래쪽은 블록 번호와 장치 I/O를 다룬다.

이 계층화 덕분에 파일 API와 실제 장치 제어를 분리할 수 있다. ==파일 시스템 구현==을 읽을 때는 이름 처리, 메타데이터 처리, 블록 배치, 장치 접근을 나눠서 보는 것이 좋다.

### ==FCB==와 ==inode==

파일 제어 블록은 파일의 메타데이터와 데이터 블록 위치 정보를 담는다. UNIX 계열에서는 ==inode==가 이 역할을 한다. 디렉터리 엔트리는 보통 파일 이름과 ==inode== 번호를 연결하고, inode는 실제 블록 위치와 권한, 크기, 시간 정보를 가진다.

이 구조를 이해하면 hard link도 자연스럽게 보인다. 여러 이름이 같은 inode를 가리킬 수 있고, inode의 link count가 0이 되면 실제 데이터 회수가 가능해진다.

> 그림 읽기: 디렉터리 엔트리는 이름을 inode에 연결하고, inode는 메타데이터와 블록 포인터를 가진다. 파일 이름과 실제 데이터 블록 사이의 간접 계층을 봐야 한다.

### 블록 할당 방식

==연속 할당==은 파일 블록을 디스크의 연속된 위치에 둔다. 순차 접근과 직접 접근이 빠르지만, 파일이 커질 때 주변에 빈 공간이 없으면 문제가 생긴다. ==연결 할당==은 각 블록이 다음 블록을 가리키므로 파일 확장은 쉽지만 임의 접근이 느리다.

==색인 할당==은 인덱스 블록이 데이터 블록 주소들을 담는다. 임의 접근과 확장에 유리하지만 인덱스 블록 비용이 든다. 실제 ==파일 시스템==은 작은 파일과 큰 파일 모두를 고려해 직접 포인터, 간접 포인터 등을 조합한다.

![파일 시스템 구현 - 파일 블록 할당 방식](/os-concepts/diagrams/os-14-file-system-implementation.png)

> 그림 읽기: ==연속 할당==은 빠르지만 성장과 외부 단편화가 문제다. ==연결 할당==은 확장이 쉽고, ==색인 할당==은 임의 접근과 큰 파일 처리에 유리하다.

### 자유 공간, 성능, 복구

새 블록을 할당하려면 빈 블록을 알아야 한다. ==bitmap==은 각 블록의 사용 여부를 비트로 표시해 단순하고 빠르게 탐색할 수 있다. free list는 빈 블록들을 연결해 관리한다. 어떤 방식이든 동시 갱신과 crash 상황을 고려해야 한다.

==파일 시스템==은 캐시와 버퍼를 통해 성능을 높인다. 하지만 쓰기 캐시가 있으면 갑작스러운 전원 차단에서 메타데이터와 데이터가 엇갈릴 수 있다. journaling과 log-structured 접근은 이런 복구 문제를 줄이기 위한 설계다.

## 작은 파일이 커질 때

1. ==연속 할당==은 처음에는 빠르지만 파일이 커질 자리가 없으면 옮기거나 새 공간을 찾아야 한다.
2. ==연결 할당==은 뒤에 블록을 붙이기 쉽지만 중간 위치로 바로 가기 어렵다.
3. ==색인 할당==은 인덱스 블록을 거쳐 데이터 블록을 찾으므로 직접 접근이 쉬워진다. inode의 직접/간접 포인터도 이 흐름에서 이해하면 된다.

## ==파일 시스템 구현==에서 자주 만나는 말들

- **==파일 시스템 계층==**: 파일 API부터 블록 I/O까지 이어지는 구현 층. 이름, 메타데이터, 블록 매핑, 장치 접근을 분리한다. 다만 파일 시스템과 장치 드라이버를 같은 층으로 보면 안 된다. 주요 키워드: `VFS`, `block`.
- **==FCB/inode==**: 파일의 메타데이터와 블록 위치를 담는 구조. 파일 이름과 실제 데이터 블록 사이를 연결한다. 다만 UNIX 계열에서는 inode 개념을 빼놓기 어렵다. 주요 키워드: `inode`, `FCB`.
- **==연속 할당==**: 파일 블록을 연속된 위치에 배치하는 방식. 순차/직접 접근이 빠르다. 다만 외부 단편화와 파일 성장 문제가 있다. 주요 키워드: `contiguous`.
- **==연결 할당==**: 각 블록이 다음 블록을 가리키는 방식. 파일 확장이 쉽고 외부 단편화가 적다. 다만 임의 접근이 느리고 포인터 손상에 취약하다. 주요 키워드: `linked`, `FAT`.
- **==색인 할당==**: 인덱스 블록이 데이터 블록 주소들을 담는 방식. 임의 접근과 파일 확장에 유리하다. 다만 작은 파일에서도 인덱스 블록 비용이 생길 수 있다. 주요 키워드: `indexed`.
- **==자유 공간 관리==**: 빈 블록을 추적하는 방식. 새 파일이나 확장에 사용할 공간을 빠르게 찾는다. 다만 ==bitmap==은 단순하지만 크기와 ==동기화== 비용이 있다. 주요 키워드: `bitmap`, `free-space`.

14장에서 한 파일 시스템 내부를 봤다면, 15장은 여러 파일 시스템을 마운트하고 공유하는 내부 구조로 넓힌다.
