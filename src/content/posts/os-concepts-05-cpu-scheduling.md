## CPU를 다음에 누구에게 줄 것인가

==프로세스==와 ==스레드==가 여러 개 있으면 이제 CPU를 누구에게 줄지 정해야 한다. 모든 작업이 CPU만 쓰는 것은 아니고, 계산하다가 I/O를 기다리고 다시 계산하는 흐름을 반복한다. 스케줄러는 ready 상태의 작업 중 하나를 골라 이 흐름을 이어 준다.

좋은 ==스케줄링==은 하나의 정답이 아니다. 짧은 응답이 중요한 시스템도 있고, 처리량이 중요한 시스템도 있으며, ==deadline==을 어기면 안 되는 시스템도 있다. ==FCFS==, ==SJF==, Round Robin, ==MLFQ==는 각각 다른 기준을 선택한 결과로 읽어야 한다.

## ==스케줄링== 기준부터 알고리즘까지 연결하기

### 좋은 스케줄러는 기준에 따라 달라진다

CPU 이용률, 처리량, 대기 시간, 반환 시간, 응답 시간은 서로 같은 방향으로 움직이지 않는다. 서버 배치 작업은 전체 처리량을 더 볼 수 있고, 대화형 프로그램은 첫 반응이 늦으면 사용자가 바로 불편함을 느낀다. ==스케줄링== 알고리즘은 정답 찾기보다 어떤 값을 포기할지 고르는 문제에 가깝다.

### CPU burst를 모르면 ==SJF==도 조심해야 한다

==SJF==는 평균 대기 시간을 줄이는 기준점으로 좋다. 다만 다음 CPU burst를 정확히 알 수 있다는 가정이 붙는다. 실제 ==운영체제==는 과거 실행 시간을 보고 다음 burst를 추정하거나, 여러 큐를 두고 실행 행동에 따라 위치를 옮기는 식으로 현실과 타협한다.

### 멀티코어에서는 CPU 선택도 ==스케줄링==이다

코어가 여러 개면 다음에 무엇을 실행할지뿐 아니라 어디에서 실행할지도 봐야 한다. 같은 코어에서 이어 실행하면 캐시가 살아 있어 유리하지만, 한 코어에 작업이 몰리면 부하 균형이 나빠진다. processor affinity와 load balancing은 이 긴장을 다루는 장치다.

### 평가 지표를 상황에 맞춰 읽기

반환 시간은 작업 제출부터 종료까지 걸린 시간이고, 대기 시간은 ==ready queue==에서 기다린 시간이며, 응답 시간은 첫 반응이 오기까지의 시간이다. 대화형 시스템에서는 응답 시간이 짧아야 하고, 배치 시스템에서는 처리량과 반환 시간이 더 중요할 수 있다. 같은 결과표도 어떤 시스템인지에 따라 다르게 읽어야 한다.

### ==선점형 스케줄링==이 ==동기화==를 어렵게 만든다

선점형 스케줄러는 실행 중인 ==프로세스==를 중간에 멈출 수 있다. 반응성에는 좋지만, 공유 자료구조를 고치는 중간에도 흐름이 끊길 수 있다는 뜻이다. 그래서 ==커널== 내부 자료구조나 멀티==스레드== 프로그램은 선점 가능성을 전제로 락과 원자 연산을 설계해야 한다.

## ==ready queue==에서 ==실시간 스케줄링==까지 따라가기

### ==스케줄링==은 ==ready queue==의 선택 문제

==프로세스==는 계속 CPU만 쓰지 않는다. 잠깐 계산하고 I/O를 기다렸다가 다시 계산하는 흐름을 반복한다. ==CPU 스케줄러==는 ready 상태의 ==프로세스== 중 하나를 골라 CPU를 넘긴다. 디스패처는 실제 문맥 교환과 사용자 모드 복귀를 수행한다.

==스케줄링==은 성능 목표에 따라 답이 달라진다. 서버는 처리량을 중시할 수 있고, 대화형 시스템은 응답 시간을 중시한다. 배치 작업은 평균 대기 시간을 보고, 실시간 시스템은 마감 시간 준수를 더 앞에 둔다.

![CPU 스케줄링 - CPU 스케줄링 위치](/os-concepts/diagrams/os-05-cpu-scheduling.png)

> 그림 읽기: 스케줄러는 ==ready queue==에서 다음 실행 대상을 고르고, dispatcher가 실제 문맥 교환을 수행한다. I/O 대기와 시간 할당량 만료가 다시 ==ready queue==로 돌아오는 대표 경로다.

![CPU 스케줄링 - 스케줄링 평가 기준](/os-concepts/diagrams/os-05-cpu-scheduling-detail.png)

> 그림 읽기: 평가 기준이 다르면 좋은 스케줄러도 달라진다. 처리량을 높이는 선택이 응답 시간을 나쁘게 만들 수 있고, ==deadline==을 지키려면 평균 성능보다 예측 가능성이 앞선다.

### 기본 알고리즘의 성격

==FCFS==는 먼저 온 작업을 먼저 처리한다. 단순하지만 긴 작업이 앞에 있으면 뒤의 짧은 작업들이 오래 기다리는 convoy effect가 생긴다. ==SJF==는 다음 CPU burst가 가장 짧은 작업을 먼저 실행해 평균 대기 시간을 줄이지만, 실제 시스템에서는 다음 burst를 정확히 알 수 없다.

Round Robin은 시간 할당량을 두고 돌아가며 실행한다. 할당량이 너무 길면 ==FCFS==처럼 되고, 너무 짧으면 문맥 교환 비용이 커진다. 그래서 ==RR==은 알고리즘 이름보다 time quantum 선택이 성능을 가른다.

### 우선순위와 기아 문제

우선순위 ==스케줄링==은 중요한 작업을 먼저 실행할 수 있게 한다. 하지만 낮은 우선순위 작업이 계속 밀리는 starvation이 생길 수 있다. ==aging==은 오래 기다린 작업의 우선순위를 점차 올려 이 문제를 줄인다.

Multilevel queue와 multilevel feedback queue는 작업 성격에 따라 큐를 나누고, 실행 행동에 따라 큐를 이동시킨다. 짧고 상호작용적인 작업은 빠르게 반응하게 하고, CPU를 오래 쓰는 작업은 낮은 큐로 내려보내는 식이다.

### 멀티코어와 실시간

코어가 여러 개면 어느 작업을 실행할지만이 아니라 어느 코어에서 실행할지도 정해야 한다. 이전에 실행된 코어의 캐시를 활용하려면 processor affinity가 유리하다. 동시에 부하 균형도 맞춰야 하므로 스케줄러의 판단이 복잡해진다.

==실시간 스케줄링==에서는 평균 응답보다 ==deadline==이 기준이 된다. Rate-monotonic은 주기가 짧은 작업에 높은 우선순위를 주고, ==EDF==는 가장 가까운 마감 시간을 가진 작업을 먼저 실행한다. 여기서 실시간은 빠르다는 뜻이 아니라 시간 제약을 예측 가능하게 지킨다는 뜻이다.

## Round Robin에서 시간 할당량을 바꿔 보면

1. 시간 할당량이 너무 길면 한 ==프로세스==가 오래 CPU를 잡아 대화형 작업의 응답이 늦어진다.
2. 너무 짧으면 문맥 교환이 자주 일어나 실제 계산보다 전환 비용이 커진다.
3. ==스케줄링== 알고리즘은 이름보다 평가 기준과 파라미터를 같이 봐야 한다.

## 스케줄링을 말할 때 필요한 기준들

- **==CPU 스케줄러==**: 준비 큐에서 다음 실행 대상을 고르는 ==커널== 구성 요소. CPU 이용률, 처리량, 응답 시간, 대기 시간을 조절한다. 다만 스케줄러가 고르고 디스패처가 실제 문맥 교환을 수행한다. 주요 키워드: `scheduler`, `ready queue`.
- **==선점형 스케줄링==**: 실행 중인 ==프로세스==를 중간에 빼앗을 수 있는 방식. 대화형 시스템의 응답성을 높인다. 다만 공유 자료구조 ==보호==를 더 신경 써야 한다. 주요 키워드: `preemption`.
- **==FCFS/SJF/RR==**: 대표적인 기본 스케줄링 알고리즘. ==FCFS==는 단순하고, ==SJF==는 평균 대기 시간을 줄이며, ==RR==은 시간 할당량으로 응답성을 만든다. 다만 ==SJF==는 다음 CPU burst를 알아야 한다는 한계가 있다. 주요 키워드: `FCFS`, `SJF`, `RR`.
- **==MLFQ==**: 여러 큐와 피드백으로 작업 성격을 반영하는 방식. 짧고 상호작용적인 작업에 빠른 응답을 주면서 긴 작업도 처리한다. 다만 큐 이동 규칙과 ==aging==을 같이 봐야 한다. 주요 키워드: `MLFQ`, `aging`.
- **==실시간 스케줄링==**: 마감 시간을 기준으로 CPU를 배정하는 방식. 평균 성능보다 ==deadline== 보장을 우선한다. 다만 실시간은 단순히 빠르다는 뜻이 아니다. 주요 키워드: `deadline`, `EDF`.

스케줄링으로 실행 순서가 섞이면 공유 데이터 접근 순서도 섞인다. 6장은 이 문제를 ==동기화==로 해결한다.
