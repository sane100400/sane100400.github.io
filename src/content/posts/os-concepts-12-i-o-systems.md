`Operating System Concepts` 10판 Chapter 12, **==I/O Systems==** 정리다.

공룡책 10판 기준 PDF p.617 근처에서 시작한다.

## 프로그램이 장치를 직접 다루지 않게 하기

프로그램이 장치를 직접 다루면 장치마다 전혀 다른 코드를 써야 하고, 잘못된 접근이 시스템 전체를 망가뜨릴 수 있다. ==운영체제==는 공통 I/O 인터페이스를 제공하고, 장치별 세부 사항은 드라이버가 맡게 한다.

I/O 장의 흐름은 CPU 낭비를 줄이는 방향으로 이어진다. ==polling==은 단순하지만 바쁘고, ==interrupt==는 완료 시점에 알려 주며, ==DMA==는 데이터 이동 자체를 CPU 밖으로 넘긴다. 버퍼와 캐시는 속도 차이를 흡수하지만, 그만큼 일관성과 복구 문제를 남긴다.

## ==interrupt==, ==DMA==, buffer를 한 그림으로 보기

### I/O 계층은 공통 인터페이스와 장치별 코드의 만남이다

응용 프로그램은 장치마다 완전히 다른 호출을 배우지 않아도 된다. ==커널== I/O 계층이 공통 인터페이스를 제공하고, ==장치 드라이버==가 실제 장치 명령을 처리한다. 이 덕분에 사용자는 파일처럼 읽고 쓰지만, 내부에서는 컨트롤러와 ==인터럽트==가 움직인다.

### ==polling==, ==interrupt==, ==DMA==는 CPU 낭비를 줄이는 방식이다

==polling==은 CPU가 장치 상태를 계속 확인하는 방식이라 단순하지만 바쁘다. ==interrupt==는 장치가 완료를 알려 주므로 CPU가 다른 일을 할 수 있다. ==DMA==는 데이터 복사 자체를 컨트롤러가 맡아 큰 전송에서 CPU 부담을 줄인다.

### 버퍼링과 캐싱은 경계를 부드럽게 만든다

장치 속도와 프로그램 속도는 잘 맞지 않는다. 버퍼는 이 속도 차이를 흡수하고, 캐시는 반복 접근을 빠르게 만든다. 다만 버퍼와 캐시가 늘면 데이터가 언제 실제 장치에 기록됐는지, 실패 시 무엇을 복구해야 하는지도 함께 생각해야 한다.

### 동기 I/O와 비동기 I/O의 차이

동기 I/O는 요청한 작업이 끝날 때까지 호출 흐름이 기다린다. 비동기 I/O는 요청을 걸어 두고 다른 일을 하다가 나중에 완료 통지를 받는다. 서버처럼 많은 연결을 다루는 프로그램에서는 이 차이가 CPU 사용률과 응답성에 큰 영향을 준다.

### 드라이버 버그가 위험한 이유

==장치 드라이버==는 하드웨어와 ==커널== 사이에 있다. 성능을 위해 ==커널== 공간에서 동작하는 경우가 많기 때문에, 드라이버 버그는 단순한 응용 프로그램 오류보다 피해가 크다. I/O 장에서 ==보호==와 안정성이 계속 나오는 이유다.

## 응용 프로그램 요청에서 장치 완료까지 따라가기

### I/O는 표준화된 인터페이스와 장치별 구현의 조합

응용 프로그램은 파일처럼 읽고 쓰는 단순한 인터페이스를 원한다. 하지만 실제 장치는 키보드, 디스크, 네트워크 카드처럼 동작 방식이 다르다. ==커널== I/O 서브시스템은 공통 인터페이스를 제공하고, ==장치 드라이버==는 특정 장치의 명령과 상태 처리를 담당한다.

==장치 컨트롤러==는 하드웨어 쪽에서 장치를 제어하고, 드라이버는 ==커널== 쪽에서 컨트롤러를 다룬다. 이 구분을 잡으면 I/O 경로가 명확해진다.

![입출력 시스템 - I/O 요청 흐름](/os-concepts/diagrams/os-12-i-o-systems.png)

> 그림 읽기: 응용 프로그램의 I/O 요청은 ==시스템 호출==, ==커널== I/O 계층, 드라이버, 컨트롤러, 장치를 차례로 지난다. ==인터럽트==와 ==DMA==는 이 경로에서 CPU 낭비를 줄이는 장치다.

### ==polling==, ==interrupt==, ==DMA==

==polling==은 CPU가 장치 상태를 반복해서 확인하는 방식이다. 구현은 단순하지만 장치가 느리면 CPU 시간이 낭비된다. ==interrupt==는 장치가 완료나 오류를 CPU에게 알려 주는 방식이다. CPU는 다른 일을 하다가 필요한 순간에 ==커널== 처리 루틴으로 이동한다.

==DMA==는 대량 데이터 전송에서 CPU가 바이트를 하나씩 복사하지 않게 한다. ==장치 컨트롤러==가 ==메모리==와 직접 데이터를 주고받고, 완료되면 ==인터럽트==로 알린다. 고성능 I/O에서 ==DMA==는 자주 등장하는 장치다.

![입출력 시스템 - DMA 전송](/os-concepts/diagrams/os-12-i-o-systems-detail.png)

> 그림 읽기: DMA를 쓰면 CPU가 바이트를 직접 복사하지 않는다. ==커널==은 버퍼와 전송 정보를 설정하고, 완료 통지는 ==인터럽트==로 받는다.

### 커널 I/O 서브시스템

커널은 I/O 요청을 단순 전달만 하지 않는다. 버퍼링으로 속도가 다른 장치 사이를 맞추고, 캐싱으로 반복 접근을 줄이며, 스풀링으로 독점 장치에 대한 요청을 큐에 쌓는다. 오류가 발생하면 재시도하거나 사용자에게 오류를 전달한다.

I/O ==보호==도 빼놓을 수 없다. 사용자 프로그램이 장치 레지스터를 직접 건드리면 시스템이 불안정해지므로, 대부분의 장치 접근은 ==시스템 호출==과 드라이버가 제한한다.

### 성능은 경로 전체에서 결정된다

I/O 요청은 응용 프로그램, ==시스템 호출==, 커널 I/O 계층, 드라이버, 컨트롤러, 장치를 거친다. 어느 한 곳만 빠르다고 전체가 빠르지는 않다. 작은 요청이 너무 자주 발생하면 ==시스템 호출==과 컨텍스트 전환 비용이 커지고, 큰 요청은 버퍼와 DMA 설정 비용을 고려해야 한다.

==운영체제==의 I/O 설계는 추상화와 성능 사이의 절충이다. 사용자는 단순한 파일 인터페이스를 원하지만, 커널은 장치 특성을 최대한 활용해야 한다.

## 디스크에서 큰 파일을 읽을 때

1. 프로그램은 read ==시스템 호출==을 보내고, 커널은 버퍼와 ==장치 드라이버==를 준비한다.
2. DMA 컨트롤러가 장치에서 ==메모리==로 데이터를 옮기는 동안 CPU는 다른 일을 할 수 있다.
3. 전송이 끝나면 ==인터럽트==가 들어오고, 커널은 대기하던 ==프로세스==에 완료를 알려 준다.

## I/O 흐름을 읽을 때 필요한 말들

- **==장치 드라이버==**: 커널과 ==장치 컨트롤러== 사이의 장치별 코드. 장치 명령과 상태 처리를 ==운영체제== 인터페이스에 맞춘다. 다만 드라이버 버그는 커널 안정성에 직접 영향을 줄 수 있다. 주요 키워드: `driver`.
- **==장치 컨트롤러==**: 실제 장치를 제어하는 하드웨어 구성 요소. 레지스터와 버퍼를 통해 CPU나 DMA와 통신한다. 다만 드라이버와 컨트롤러는 소프트웨어/하드웨어로 구분된다. 주요 키워드: `controller`.
- **==폴링==**: CPU가 장치 상태를 반복 확인하는 방식. 단순한 장치나 짧은 대기에서 쓸 수 있다. 다만 대기 시간이 길면 CPU를 낭비한다. 주요 키워드: `polling`.
- **==인터럽트==**: 장치가 CPU에게 처리 필요를 알리는 방식. CPU가 I/O 완료를 기다리며 놀지 않게 한다. 다만 너무 잦으면 ==interrupt== overhead가 커진다. 주요 키워드: `interrupt`.
- **DMA**: 장치와 ==메모리== 사이의 데이터 전송을 CPU 대신 처리하는 방식. 대량 I/O에서 CPU 복사 부담을 줄인다. 다만 ==메모리== ==보호==와 캐시 일관성을 함께 봐야 한다. 주요 키워드: `DMA`.

I/O가 블록을 읽고 쓰는 경로라면, 13장은 사용자가 보는 파일과 디렉터리 인터페이스를 다룬다.
