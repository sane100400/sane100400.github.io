## 블록 번호 대신 ==파일== 이름으로 생각하기

저장장치를 그대로 블록 번호로 다루면 사람도 프로그램도 쓰기 어렵다. ==파일 시스템 인터페이스==는 저장된 데이터를 이름, ==디렉터리==, ==파일== 디스크립터 같은 추상화로 감싼다. 사용자는 경로를 열고 읽고 쓰지만, ==커널==은 열린 ==파일== 테이블과 offset을 관리한다.

이 장의 흐름은 이름에서 열린 상태로 넘어가는 과정이다. 경로는 ==파일==을 찾는 데 필요하고, 한 번 열고 나면 ==파일== 디스크립터가 중요해진다. 순차 접근과 직접 접근, ==디렉터리== 구조, 링크는 모두 ==파일==을 어떤 방식으로 찾고 사용할지에 대한 선택이다.

## ==디렉터리==, ==파일== 디스크립터, offset 연결하기

### ==파일==은 블록 장치를 사람이 쓰기 좋게 바꾼 이름이다

사용자는 디스크의 몇 번째 블록을 읽겠다고 하지 않는다. 이름이 붙은 파일을 열고, 읽고, 쓰고, 닫는다. ==운영체제==는 이 요청을 내부의 파일 제어 블록, 열린 파일 테이블, 디스크 블록 접근으로 바꾼다.

### 경로와 파일 디스크립터는 역할이 다르다

경로는 파일을 찾기 위한 이름이다. 한 번 `open()`이 성공하면 프로그램은 보통 경로 대신 파일 디스크립터를 쓴다. 파일 디스크립터는 열린 파일 테이블의 항목을 가리키고, 그 항목에는 현재 offset과 접근 모드가 들어 있다.

### ==접근 방법==은 파일의 사용 방식을 반영한다

로그 파일처럼 앞에서부터 읽는 데이터는 순차 접근이 자연스럽다. 데이터베이스처럼 중간 위치를 바로 찾아야 하는 경우에는 직접 접근이 필요하다. 파일 인터페이스는 단순해 보여도 내부 구현과 캐싱 정책은 이 사용 방식에 영향을 받는다.

### 열린 파일 테이블을 봐야 offset이 이해된다

파일 이름은 디스크의 파일을 찾는 데 쓰이고, 파일을 연 뒤에는 열린 파일 테이블 항목이 현재 읽기 위치와 접근 모드를 기억한다. 같은 파일을 두 번 열면 서로 다른 offset을 가질 수 있고, fork 뒤에는 부모와 자식이 같은 열린 파일 항목을 공유할 수 있다.

### 하드 링크와 심볼릭 링크의 차이

하드 링크는 같은 파일 객체에 이름을 하나 더 붙이는 방식에 가깝고, 심볼릭 링크는 다른 경로를 가리키는 별도의 파일이다. 그래서 하드 링크는 대상 파일과 inode를 공유하지만, 심볼릭 링크는 경로 해석을 한 번 더 거친다. ==디렉터리==와 이름 공간을 이해하는 좋은 예다.

## 파일 이름에서 열린 파일 테이블까지 따라가기

### 파일은 저장장치의 사용자 친화적 추상화

사용자는 디스크 블록 번호를 직접 기억하지 않는다. ==파일 시스템==은 데이터를 이름 있는 파일로 보여 주고, 파일에 대한 open, read, write, close 같은 연산을 제공한다. 파일에는 내용뿐 아니라 이름, 식별자, 위치, 크기, ==보호== 정보, 시간 정보 같은 메타데이터가 붙는다.

열린 파일은 파일 이름보다 파일 디스크립터나 핸들로 다뤄진다. 이름은 찾기 위한 정보이고, 열린 뒤에는 ==커널==이 관리하는 열린 파일 테이블의 항목이 중요해진다.

![파일 시스템 인터페이스 - 열린 파일 관리](/os-concepts/diagrams/os-13-file-system-interface-detail.png)

> 그림 읽기: ==프로세스==의 파일 디스크립터는 열린 파일 테이블을 가리키고, 그 항목은 offset과 접근 모드를 가진다. 같은 파일도 열린 상태에 따라 읽는 위치가 달라질 수 있다.

### ==접근 방법==

순차 접근은 앞에서부터 차례대로 읽거나 쓰는 방식이다. 로그 파일이나 스트림 처리에 잘 맞는다. 직접 접근은 특정 위치의 블록이나 레코드로 바로 이동해 읽는 방식이다. 데이터베이스나 큰 파일의 일부만 수정하는 작업에 필요하다.

==접근 방법==은 ==파일 시스템== 내부 구현과 캐싱 정책에도 영향을 준다. ==운영체제==는 순차 접근에서는 미리 읽기를 할 수 있고, 랜덤 접근에서는 다른 최적화가 필요하다.

### ==디렉터리==와 경로

==디렉터리==는 파일 이름을 파일의 내부 식별자나 메타데이터로 연결하는 자료구조다. 단일 ==디렉터리==는 단순하지만 이름 충돌과 관리 문제가 크다. 트리 구조 ==디렉터리==는 사용자별, 프로젝트별로 파일을 나눠 관리하기 쉽다.

경로 해석은 루트 또는 현재 디렉터리에서 시작해 각 이름 요소를 차례대로 찾는 과정이다. 심볼릭 링크가 있으면 경로 해석이 다른 경로로 이어질 수 있다. 따라서 디렉터리는 화면의 폴더 아이콘이 아니라 이름 공간을 관리하는 기준이 되는 구조다.

![파일 시스템 인터페이스 - 경로 이름 해석](/os-concepts/diagrams/os-13-file-system-interface.png)

> 그림 읽기: 경로 문자열은 디렉터리를 따라 inode나 FCB를 찾기 위한 입력이다. 파일을 연 뒤에는 경로보다 파일 디스크립터와 열린 파일 테이블 항목이 중요해진다.

### ==파일 보호==

여러 사용자와 ==프로세스==가 같은 ==파일 시스템==을 쓰면 권한이 필요하다. 읽기, 쓰기, 실행, 삭제, 소유자 변경 같은 연산을 사용자나 그룹별로 제한한다. UNIX의 rwx 비트는 단순한 모델이고, ==ACL==은 객체별로 더 세밀한 권한 목록을 둘 수 있다.

==파일 보호==는 16장 ==보안==과 17장 ==보호==의 구체적인 사례다. 이름을 찾을 수 있다는 것과 내용을 읽을 수 있다는 것은 다른 문제이며, 실행 권한도 읽기 권한과 별개로 판단해야 한다.

## 파일 이름과 열린 파일의 차이

1. 프로그램이 `/home/a.txt`를 열 때는 디렉터리를 따라가며 이름을 inode나 FCB로 해석한다.
2. 한 번 열고 나면 프로그램은 경로 문자열 대신 파일 디스크립터를 사용한다.
3. 파일 디스크립터는 열린 파일 테이블을 가리키고, 그 안에는 현재 offset과 접근 모드가 저장된다.

## 파일 인터페이스를 설명하는 말들

- **파일**: 이름이 붙은 데이터 저장 단위. 바이트 배열 또는 레코드 집합을 일관된 방식으로 읽고 쓴다. 다만 파일 이름은 경로 해석의 입력이고, 열린 뒤에는 파일 디스크립터를 봐야 한다. 주요 키워드: `file`, `descriptor`.
- **==파일 속성==**: 이름, 식별자, 위치, 크기, 권한, 시간 정보. 파일 관리와 ==보호==에 필요한 메타데이터를 제공한다. 다만 속성은 파일 내용과 별개로 관리된다. 주요 키워드: `metadata`.
- **==접근 방법==**: 파일 데이터를 읽는 방식. 순차 접근은 스트림에, 직접 접근은 블록/레코드 기반 처리에 적합하다. 다만 접근 패턴은 ==파일 시스템== 성능에 영향을 준다. 주요 키워드: `sequential`, `direct`.
- **디렉터리**: 이름을 파일 메타데이터에 연결하는 자료구조. 경로 탐색, 파일 그룹화, 이름 충돌 방지를 담당한다. 다만 폴더 UI가 아니라 이름 매핑 구조로 이해해야 한다. 주요 키워드: `directory`, `path`.
- **==파일 보호==**: 파일 연산 권한을 통제하는 방식. read, write, execute, delete 등을 사용자나 그룹별로 제한한다. 다만 실행 권한과 읽기 권한은 같은 뜻이 아니다. 주요 키워드: `permission`, `ACL`.

13장이 사용자에게 보이는 인터페이스라면, 14장은 그 파일을 실제 블록에 배치하는 구현을 본다.
