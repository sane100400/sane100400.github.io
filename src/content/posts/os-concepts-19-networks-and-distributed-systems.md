공룡책 10판 기준 PDF p.871 근처에서 시작한다.

## 컴퓨터가 여러 대가 되면 실패도 여러 모양이 된다

한 컴퓨터 안의 자원 관리가 어느 정도 정리되면, 여러 컴퓨터가 네트워크로 협력할 때의 문제가 보인다. ==분산 시스템==은 자원 공유와 확장성을 주지만, 실패가 더 복잡해진다. 전체가 멈추는 대신 일부 노드만 느리거나 응답이 사라질 수 있다.

이 장의 흐름은 통신, 이름, 파일, 실패 처리로 이어진다. ==TCP==와 ==UDP==는 통신의 성격을 정하고, 이름 서비스는 위치를 숨기며, ==분산 파일 시스템==은 원격 데이터를 로컬처럼 보이게 한다. 그 대가로 캐시 일관성과 partial failure를 계속 다뤄야 한다.

## 통신, 이름, 캐시, partial failure 연결하기

### ==분산 시스템==은 부분 실패를 기본값으로 둔다

한 컴퓨터 안에서는 CPU나 ==메모리==가 실패하면 대개 시스템 전체가 멈춘다. ==분산 시스템==에서는 서버 하나만 죽거나, 네트워크만 느려지거나, 응답만 유실될 수 있다. 그래서 클라이언트는 느린 것과 실패한 것을 구분하기 어렵다.

### 이름은 위치를 숨기는 약속이다

==DNS==가 이름을 IP 주소로 바꾸듯, ==분산 파일 시스템==도 파일 이름을 실제 서버와 객체 위치로 연결한다. 이름과 위치를 분리하면 이동과 복제가 쉬워진다. 대신 이름 서비스와 캐시가 오래된 정보를 줄 수 있다는 새 문제가 생긴다.

### 일관성은 로컬 파일보다 비싸다

여러 클라이언트가 같은 파일을 캐시하고 수정하면 누가 최신 값을 보는지 정해야 한다. 강한 일관성을 원하면 통신이 많아지고 지연이 늘 수 있다. 느슨한 일관성을 허용하면 빠르지만 사용자가 잠시 다른 값을 볼 수 있다.

### 재시도는 항상 안전하지 않다

==분산 시스템==에서 응답이 없다고 같은 요청을 다시 보내면 중복 실행이 생길 수 있다. 읽기 요청은 대체로 안전하지만, 결제나 파일 쓰기 같은 요청은 한 번만 실행되어야 할 수 있다. 그래서 request id, idempotent operation, 로그 기반 복구 같은 장치가 필요하다.

### 일관성 모델을 먼저 정해야 한다

모든 클라이언트가 항상 최신 값을 보게 만들면 이해하기 쉽지만 통신 비용이 커진다. 반대로 eventual consistency를 허용하면 빠르고 견고해질 수 있지만 잠시 다른 값을 보는 상황을 받아들여야 한다. ==분산 시스템==은 성능과 가용성, 일관성의 균형을 계속 묻는다.

## ==프로토콜==에서 ==분산 파일 시스템==까지 따라가기

### ==분산 시스템==의 동기

==분산 시스템==은 여러 컴퓨터를 네트워크로 연결해 자원 공유, 계산 분산, 신뢰성 향상, 확장성을 얻으려는 구조다. 사용자는 하나의 서비스처럼 느끼지만, 실제로는 여러 노드가 메시지를 주고받으며 협력한다.

==운영체제== 관점에서는 로컬 자원만 관리하던 문제가 네트워크 너머로 확장된다. 파일, ==프로세스==, 이름, 인증, 장애 처리가 모두 더 복잡해진다.

![네트워크와 분산 시스템 - 분산 시스템 실패](/os-concepts/diagrams/os-19-networks-and-distributed-systems-detail.png)

> 그림 읽기: ==분산 시스템==의 실패는 전부 멈추는 형태만이 아니다. 요청 지연, 패킷 유실, 서버 장애가 섞이면 클라이언트는 느린 것과 실패한 것을 구분하기 어렵다.

### 통신과 ==프로토콜==

네트워크 통신은 정해진 ==프로토콜==에 따라 이루어진다. ==TCP==는 연결 기반의 신뢰성 있는 바이트 스트림을 제공하고, ==UDP==는 더 단순한 datagram 전달을 제공한다. 어떤 ==프로토콜==을 쓸지는 신뢰성, 지연, 순서 보장, 구현 복잡도에 따라 달라진다.

==운영체제==는 소켓 인터페이스를 통해 네트워크 통신을 파일과 비슷한 방식으로 사용할 수 있게 한다. 하지만 네트워크는 로컬 디스크보다 실패 양상이 다양하다.

### 이름과 위치

분산 시스템에서는 사람이 기억하는 이름을 실제 위치나 주소로 바꾸는 과정이 필요하다. ==DNS==는 도메인 이름을 IP 주소로 바꾸고, ==분산 파일 시스템==은 파일 이름을 서버와 객체 위치로 연결한다. 이름과 위치를 분리하면 이동, 복제, 부하 분산이 쉬워진다.

하지만 이름 서비스 자체도 장애와 일관성 문제를 가진다. 캐시는 성능을 높이지만 오래된 정보를 제공할 수 있다.

### ==분산 파일 시스템==

==분산 파일 시스템==은 원격 파일을 로컬 파일처럼 접근하게 한다. 클라이언트는 경로를 해석하고, 서버에 요청을 보내고, 결과를 캐시할 수 있다. 사용자는 편리하지만, 여러 클라이언트가 같은 파일을 수정할 때 일관성 문제가 생긴다.

분산 시스템의 가장 까다로운 점은 partial failure다. 한 노드는 살아 있고 다른 노드는 죽을 수 있으며, 네트워크만 끊길 수도 있다. 로컬 ==운영체제==에서는 드문 종류의 불확실성이 기본 상황이 된다.

![네트워크와 분산 시스템 - 분산 파일 시스템](/os-concepts/diagrams/os-19-networks-and-distributed-systems.png)

> 그림 읽기: ==분산 파일 시스템==은 이름 서비스, 서버 저장소, 클라이언트 캐시가 함께 움직인다. 캐시는 빠르지만 여러 클라이언트가 같은 파일을 볼 때 일관성 문제가 생긴다.

## 분산 파일 요청이 실패했는지 느린지 모를 때

1. 클라이언트가 서버에 요청을 보냈는데 응답이 없으면 서버가 죽었는지, 네트워크가 느린지, 응답만 유실됐는지 바로 알기 어렵다.
2. 그래서 timeout, retry, idempotent operation, 캐시 무효화 같은 정책이 필요하다.
3. 분산 시스템의 어려움은 여러 노드가 있다는 사실보다 부분 실패를 다뤄야 한다는 데 있다.

## 분산 시스템에서 필요한 말들

- **분산 시스템**: 네트워크로 연결된 여러 컴퓨터가 협력하는 시스템. 자원 공유, 확장성, 신뢰성 향상을 노린다. 다만 부분 실패가 가장 까다로운 점이다. 주요 키워드: `distributed system`.
- **==이름 해석==**: 사람이 쓰는 이름을 주소나 위치로 바꾸는 과정. 도메인 이름, 파일 경로, 서비스 이름을 실제 위치와 연결한다. 다만 이름과 위치를 분리해야 이동성과 복제가 쉬워진다. 주요 키워드: `naming`, `DNS`.
- **==프로토콜==**: 통신 주체가 메시지를 주고받는 규칙. 오류 처리, 순서, 흐름 제어, 연결 관리를 정의한다. 다만 ==TCP==와 ==UDP==는 신뢰성 제공 방식이 다르다. 주요 키워드: `protocol`, `TCP`, `UDP`.
- **==투명성==**: 분산되어 있다는 사실을 사용자에게 덜 보이게 하는 성질. 위치, 복제, 장애, 이동을 숨긴다. 다만 완전한 ==투명성==은 성능과 복잡도 비용을 만든다. 주요 키워드: `transparency`.
- **==분산 파일 시스템==**: 원격 파일을 로컬 파일처럼 다루는 시스템. 클라이언트 캐시와 서버 저장소를 조합한다. 다만 성능과 일관성 사이의 절충을 봐야 한다. 주요 키워드: `DFS`, `cache`.

이제 추상 개념을 실제 ==운영체제== 사례로 본다. 20장은 ==Linux==가 앞 장의 개념을 어떻게 구현하는지 정리한다.
