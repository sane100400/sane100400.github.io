## 여러 ==파일 시스템==을 하나의 트리처럼 보이게 하기

한 컴퓨터에는 여러 종류의 ==파일 시스템==이 함께 있을 수 있다. 로컬 디스크, 임시 ==파일 시스템==, ==커널== 상태를 보여 주는 가상 ==파일 시스템==, ==원격 파일 시스템==이 모두 같은 경로처럼 보인다. 이 통일된 이름 공간을 만들기 위해 ==마운트==와 ==VFS==가 필요하다.

==VFS==는 서로 다른 구현을 같은 open, read, write 흐름으로 묶는다. ==원격 파일 시스템==까지 들어오면 네트워크 실패와 캐시 일관성 문제가 추가된다. ==파일 시스템 내부== 장은 저장장치와 ==메모리==, 네트워크가 파일이라는 추상화 아래에서 만나는 지점이다.

## 로컬 파일과 원격 파일을 같은 호출로 다루기

### ==마운트==는 파일을 복사하는 일이 아니다

==마운트==는 특정 디렉터리 아래의 경로 해석을 다른 ==파일 시스템== 루트로 넘기는 연결이다. USB 드라이브나 ==원격 파일 시스템==을 붙이면 사용자는 같은 경로 문법으로 접근하지만, 내부 구현은 완전히 다를 수 있다.

### ==VFS==는 여러 ==파일 시스템==을 같은 모양으로 보이게 한다

ext4, XFS, tmpfs, ==NFS==는 내부 구현이 다르다. 그래도 응용 프로그램은 `open`, `read`, `write` 같은 호출을 쓴다. ==VFS==는 공통 객체와 연산 집합을 제공해 이 차이를 ==커널== 안에서 흡수한다.

### 원격 파일은 캐시와 일관성이 늘 따라온다

분산 ==파일 시스템==은 원격 서버의 파일을 로컬 파일처럼 보이게 한다. 클라이언트 캐시는 성능을 올리지만 여러 클라이언트가 같은 파일을 바꿀 때 오래된 데이터를 볼 수 있다. ==파일 시스템 내부== 장은 결국 성능과 일관성의 타협을 계속 묻는다.

### page cache는 ==메모리==와 ==파일 시스템==을 이어 준다

파일을 읽을 때 디스크 블록은 보통 page cache에 올라온다. 이후 같은 파일을 다시 읽으면 디스크를 가지 않고 ==메모리==에서 끝날 수 있다. 이 때문에 ==메모리== 관리와 파일 시스템 성능은 따로 떨어져 있지 않다. ==메모리==가 부족하면 page cache를 얼마나 버릴지도 정책 문제가 된다.

### ==원격 파일 시스템==은 실패 의미부터 다르다

로컬 디스크는 느리더라도 대개 응답이 오거나 오류가 난다. ==원격 파일 시스템==에서는 요청이 사라졌는지, 서버가 처리했지만 응답만 유실됐는지 알기 어렵다. retry가 안전하려면 연산이 idempotent한지, 캐시를 언제 무효화할지 같은 규칙이 필요하다.

## ==VFS==에서 ==NFS== 일관성까지 따라가기

### ==마운트==는 이름 공간에 파일 시스템을 붙이는 일

==운영체제==는 여러 저장장치와 여러 파일 시스템을 하나의 디렉터리 트리처럼 보여 줄 수 있다. ==마운트==는 특정 장치나 ==원격 파일 시스템==의 루트를 기존 디렉터리 위치에 연결하는 작업이다. 사용자는 같은 경로 탐색 방식으로 접근하지만, 내부적으로는 다른 파일 시스템 코드가 동작할 수 있다.

==마운트==는 파일 복사가 아니다. 이름 공간에서 어느 지점 아래의 경로 해석을 다른 파일 시스템으로 넘기는 연결 작업에 가깝다.

![파일 시스템 내부 - 마운트와 VFS](/os-concepts/diagrams/os-15-file-system-internals-detail.png)

> 그림 읽기: ==마운트==는 데이터를 복사하는 일이 아니라 경로 해석의 특정 지점을 다른 파일 시스템 루트와 연결하는 일이다.

### ==VFS==가 필요한 이유

한 시스템에는 ext4, XFS, tmpfs, procfs, ==NFS==처럼 성격이 다른 파일 시스템이 함께 존재할 수 있다. 응용 프로그램이 파일 시스템마다 다른 API를 배워야 한다면 불편하다. ==VFS==는 공통 vnode/inode 추상화를 제공해 open, read, write 같은 호출을 적절한 파일 시스템 구현으로 분배한다.

==VFS==는 실제 디스크 형식이 아니다. 서로 다른 구현을 같은 인터페이스로 보이게 하는 ==커널== 내부 계층이다.

![파일 시스템 내부 - VFS 계층](/os-concepts/diagrams/os-15-file-system-internals.png)

> 그림 읽기: VFS는 ext4, XFS, ==NFS==, procfs처럼 서로 다른 구현을 같은 open/read/write 인터페이스로 묶는다. 응용 프로그램은 파일 시스템 종류를 매번 의식하지 않는다.

### ==파일 공유==와 일관성

여러 ==프로세스==나 사용자가 같은 파일을 읽고 쓰면 변경 내용이 언제 보이는지가 문제가 된다. UNIX semantics는 쓰기가 즉시 다른 reader에게 보이는 모델에 가깝고, session semantics는 파일을 닫는 시점에 변경이 반영되는 식의 느슨한 모델을 허용한다.

일관성을 강하게 유지하면 구현과 네트워크 비용이 커진다. 반대로 캐시를 적극적으로 쓰면 성능은 좋아지지만 오래된 데이터를 볼 수 있다. ==파일 공유==는 성능과 일관성의 절충이다.

### ==원격 파일 시스템==

==NFS== 같은 ==원격 파일 시스템==은 네트워크 너머의 파일을 로컬 파일처럼 접근하게 한다. 클라이언트는 파일 연산을 ==RPC==로 서버에 보내고, 서버는 저장장치의 파일 시스템을 조작해 결과를 돌려준다. 클라이언트 캐시는 성능을 높이지만 일관성 문제를 만든다.

네트워크는 실패할 수 있다. 서버가 재시작되거나 패킷이 유실되어도 파일 시스템은 가능한 한 예측 가능한 동작을 제공해야 한다. 이 점이 로컬 파일 시스템과 원격 파일 시스템의 큰 차이다.

## ==NFS== 파일을 로컬 파일처럼 읽을 때

1. 응용 프로그램은 로컬 파일과 같은 `open`, `read` 호출을 사용한다.
2. VFS는 경로가 원격 파일 시스템에 걸려 있음을 보고 ==NFS== 클라이언트 코드로 요청을 넘긴다.
3. 서버 응답은 클라이언트 캐시에 남을 수 있다. 이 캐시가 성능을 높이지만 일관성 문제도 만든다.

## ==파일 시스템 내부==를 읽는 말들

- **==마운트==**: 다른 파일 시스템을 기존 디렉터리 트리에 붙이는 작업. 여러 장치와 원격 파일 시스템을 하나의 이름 공간으로 보이게 한다. 다만 마운트는 파일 복사가 아니다. 주요 키워드: `mount`.
- **VFS**: 서로 다른 파일 시스템을 공통 인터페이스로 감싸는 계층. ext4, XFS, ==NFS==, procfs를 같은 파일 API로 다루게 한다. 다만 VFS는 실제 저장 형식이 아니라 추상화 계층이다. 주요 키워드: `VFS`.
- **==파일 공유==**: 여러 사용자나 ==프로세스==가 같은 파일에 접근하는 상황. 권한, 잠금, 동시 갱신 문제를 다룬다. 다만 공유 파일은 ==일관성 의미론==을 함께 봐야 한다. 주요 키워드: `sharing`, `lock`.
- **원격 파일 시스템**: 네트워크 너머 파일을 로컬처럼 접근하는 방식. NFS처럼 클라이언트-서버 구조로 파일 연산을 제공한다. 다만 네트워크 실패와 캐시 일관성이 주요 문제다. 주요 키워드: `NFS`, `RPC`.
- **==일관성 의미론==**: 한 사용자의 변경이 다른 사용자에게 언제 보이는지에 대한 규칙. UNIX semantics, session semantics 등을 구분한다. 다만 성능을 위해 캐시하면 즉시 일관성은 어려워진다. 주요 키워드: `consistency`.

파일과 자원 공유를 봤으니, 16장부터는 공격과 방어라는 ==보안== 관점으로 넘어간다.
