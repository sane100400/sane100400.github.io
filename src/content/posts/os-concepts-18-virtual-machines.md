`Operating System Concepts` 10판 Chapter 18, **==Virtual Machines==** 정리다.

공룡책 10판 기준 PDF p.838 근처에서 시작한다.

## 컴퓨터 한 대 안에 컴퓨터를 또 만들기

==프로세스==는 같은 ==커널==을 공유하는 격리 단위다. ==가상 머신==은 한 단계 더 나아가 guest OS까지 포함한 실행 환경 전체를 격리한다. 게스트는 자신이 실제 하드웨어를 쓰는 것처럼 행동하지만, 실제 자원 배분은 ==하이퍼바이저==가 맡는다.

그래서 ==가상 머신== 장은 CPU, ==메모리==, I/O를 한 번 더 속이는 방법을 다룬다. 특권 명령을 안전하게 처리하고, guest physical address를 host physical address로 다시 매핑하며, 장치 접근을 에뮬레이션하거나 우회한다. 클라우드와 테스트 환경의 기반이 이 구조다.

## CPU, ==메모리==, I/O 가상화를 나눠 보기

### ==가상 머신==은 실행 환경 전체를 감싼다

==프로세스==는 같은 ==커널==을 공유하지만, ==가상 머신==은 guest OS까지 포함한 환경을 분리한다. 게스트는 자신이 하드웨어를 직접 쓰는 것처럼 행동하고, ==하이퍼바이저==가 실제 CPU, ==메모리==, I/O 자원을 나눠 준다. 그래서 테스트, 서버 통합, 클라우드에서 자주 쓰인다.

### 특권 명령을 안전하게 처리해야 한다

guest OS도 ==운영체제==라서 특권 명령을 실행하려 한다. 하지만 실제 하드웨어를 마음대로 건드리면 다른 ==VM==을 망가뜨릴 수 있다. ==하이퍼바이저==는 ==trap-and-emulate==, ==binary translation==, 하드웨어 지원 가상화 같은 방법으로 이 요청을 가로채고 처리한다.

### ==메모리==와 I/O는 한 층 더 복잡해진다

게스트가 보는 물리 주소는 호스트의 실제 물리 주소가 아니다. 주소 변환이 한 단계 더 생기고, 장치 I/O도 에뮬레이션이나 paravirtualized driver, passthrough 같은 선택지가 생긴다. ==VM== 성능을 볼 때 CPU만 보면 부족한 이유다.

### 중첩 주소 변환이 성능을 좌우한다

게스트 ==프로세스==의 주소는 guest virtual address이고, 게스트 OS는 이를 guest physical address로 바꾼다. ==하이퍼바이저==는 다시 host physical address로 매핑한다. 하드웨어 지원이 없으면 이 변환 비용이 크다. EPT나 nested paging 같은 기능은 이 비용을 줄이기 위해 등장했다.

### I/O 가상화는 호환성과 성능의 거래다

장치를 완전히 에뮬레이션하면 guest OS가 기존 드라이버를 그대로 쓸 수 있지만 느릴 수 있다. paravirtualized driver는 게스트가 가상화 환경을 알고 협력하므로 빠르지만 전용 드라이버가 필요하다. passthrough는 성능이 좋지만 이동성과 공유가 어려워질 수 있다.

## ==하이퍼바이저==에서 ==live migration==까지 따라가기

### ==가상 머신==의 기본 생각

==가상 머신==은 하나의 물리 컴퓨터 위에 여러 개의 독립된 실행 환경을 만든다. 각 guest OS는 자신이 실제 하드웨어를 가진 것처럼 동작하지만, 실제 자원 배분은 ==하이퍼바이저==가 조정한다. 이 구조는 격리, 서버 통합, 테스트, 클라우드 배포에 유용하다.

==가상 머신==은 ==프로세스==보다 큰 격리 단위다. ==프로세스==는 같은 ==커널==을 공유하지만, ==VM==은 guest OS까지 포함한 독립 환경을 제공한다.

### ==하이퍼바이저==의 종류

Type 1 ==하이퍼바이저==는 하드웨어 위에서 직접 실행된다. 서버 가상화와 클라우드 환경에서 많이 쓰인다. Type 2 하이퍼바이저는 호스트 ==운영체제== 위의 응용 프로그램처럼 실행되며, 데스크톱 개발과 테스트 환경에서 접근하기 쉽다.

구분은 배치 구조의 차이다. 어느 쪽이 항상 좋다고 외우기보다, 성능, 관리, 장치 지원, 사용 목적에 따라 선택한다고 보는 편이 정확하다.

![가상 머신 - Type 1과 Type 2 하이퍼바이저](/os-concepts/diagrams/os-18-virtual-machines.png)

> 그림 읽기: Type 1은 하드웨어 위에 바로 놓이고 Type 2는 호스트 OS 위에서 실행된다. 둘의 차이는 ==가상 머신==을 어느 계층에서 관리하느냐에 있다.

### CPU와 ==메모리== 가상화

CPU 가상화에서는 guest OS가 특권 명령을 실행하려 할 때 하이퍼바이저가 이를 안전하게 처리해야 한다. ==trap-and-emulate==, ==binary translation==, 하드웨어 지원 가상화는 이 문제를 푸는 방법들이다. 현대 CPU는 가상화를 위한 명령과 모드를 제공한다.

==메모리== 가상화에서는 guest virtual address, guest physical address, host physical address가 구분된다. 하이퍼바이저는 guest가 보는 물리 ==메모리==를 실제 물리 ==메모리==에 매핑해야 한다. 이 과정에서도 TLB와 페이지 테이블 비용을 봐야 한다.

![가상 머신 - 메모리 가상화 주소](/os-concepts/diagrams/os-18-virtual-machines-detail.png)

> 그림 읽기: 게스트가 보는 물리 주소와 호스트의 실제 물리 주소는 다르다. 하이퍼바이저 매핑이 한 층 더 들어가면서 주소 변환 비용도 늘어난다.

### I/O와 운영 기능

I/O 가상화는 장치가 다양하고 성능 요구가 높아 까다롭다. 완전 에뮬레이션은 호환성이 좋지만 느릴 수 있고, paravirtualized driver나 device passthrough는 성능을 높이지만 제약이 생긴다.

스냅샷은 특정 시점의 ==VM== 상태를 저장하고 되돌릴 수 있게 한다. ==Live migration==은 실행 중인 ==VM==을 다른 물리 호스트로 옮겨 유지보수와 부하 분산을 가능하게 한다. 이 기능들은 클라우드 인프라의 기반이다.

## 게스트 OS가 메모리를 보는 방식

1. 게스트 ==프로세스==는 guest virtual address를 만든다. 게스트 OS는 이를 guest physical address로 바꾼다.
2. 하이퍼바이저는 guest physical address를 실제 host physical address로 다시 매핑한다.
3. 주소 변환이 한 층 더 생기므로 하드웨어 지원과 TLB 관리가 성능에 큰 영향을 준다.

## 가상 머신을 설명하는 말들

- **가상 머신**: 소프트웨어로 만든 독립 실행 환경. 하나의 물리 장치 위에서 여러 OS나 실행 환경을 격리한다. 다만 ==VM==은 컨테이너보다 아래 계층, 즉 하드웨어 가상화에 가깝다. 주요 키워드: `VM`.
- **하이퍼바이저**: 가상 머신을 만들고 물리 자원을 나눠 주는 계층. CPU, 메모리, I/O, 저장장치를 가상화한다. 다만 Type 1은 하드웨어 위, Type 2는 호스트 OS 위에서 동작한다. 주요 키워드: `hypervisor`.
- **==Trap-and-emulate==**: 민감한 명령을 잡아 하이퍼바이저가 대신 처리하는 방식. 게스트 OS가 직접 하드웨어를 만지는 것처럼 보이게 한다. 다만 모든 민감 명령이 자동으로 ==trap==되는 것은 아니다. 주요 키워드: `trap`.
- **==Binary translation==**: 문제가 되는 명령을 안전한 명령열로 바꾸는 방식. 하드웨어 지원이 부족한 환경에서 가상화를 가능하게 한다. 다만 성능 비용이 있다. 주요 키워드: `binary translation`.
- **==Live migration==**: 실행 중인 ==VM==을 다른 물리 호스트로 옮기는 기술. 서비스 중단을 줄이며 유지보수와 부하 분산을 돕는다. 다만 메모리 변경분과 I/O 상태 처리가 어렵다. 주요 키워드: `live migration`.

한 컴퓨터 안의 격리를 봤다면, 19장은 여러 컴퓨터가 네트워크로 협력할 때 생기는 ==운영체제== 문제를 본다.
